## Test Agent drawer

Open any agent, click **Test Agent** in the header, and keep the drawer pinned to the right side of the editor. The drawer exposes two tabs—**Web Call** and **Phone Call**—plus all the inputs your team needs to configure a realistic scenario. Staying inside this drawer keeps prompt tweaks and QA runs in the same place.

## Configure a web call

<Steps>
  <Step title="Open the agent">
    Go to **AI Agents → Select Agent** and ensure the latest instructions are saved.
  </Step>
  <Step title="Choose Web Call">
    In the Test Agent drawer, keep **Web Call** selected in the segmented control.
  </Step>
  <Step title="Set direction">
    Pick **Inbound** to pretend a caller is dialing in or **Outbound** to emulate the agent calling a contact. This value is injected into the real-time calling service context when a context endpoint exists.
  </Step>
  <Step title="Provide contact data">
    Add a phone number if your instructions rely on `contact.phone_number` or if your Dynamic Context endpoint branches on it. Leave it blank for purely prompt-driven tests.
  </Step>
  <Step title="Start the session">
    Press **Start Test**. The drawer closes automatically and the floating call widget appears so you can monitor the run.
  </Step>
</Steps>

## Working with test variables

- All variables defined on the agent appear once (duplicates by UUID are deduplicated in the UI).
- Update values inline and click **Save Changes** to persist them via `useUpdateAgentVariables`.
- Saved defaults travel with the agent so the next tester sees the same presets.
- Use clear, self-describing sample values ("VIP caller", "Trial plan") so everyone can reproduce the scenario quickly.

## Context-aware testing

If the agent has a **Context API endpoint**, the drawer sends a `testContext` payload whenever you fill the optional contact and agent numbers. Use this to:

- Simulate CRM lookups without deploying a campaign
- Test different routing branches by changing `direction`
- Verify the backend receives the metadata it expects before you touch the phone network

## Floating call widget

The widget that appears after you click **Start Test** gives you live call controls:

- Shows connection status (Connecting, Listening, Speaking, or On Call)
- Displays live transcription with speaker labels and function-call dumps for tool executions
- Lets you minimize or end the call without reopening the drawer
- Keeps running even if you navigate to a different page so you can inspect other screens during a call

Use it to capture screenshots or note-taking timestamps while the call is in progress.

## After the run

1. Go to **Conversations → Recent**.
2. Filter by the agent you just tested.
3. Open the latest entry, replay audio if needed, and confirm goal tracking plus action payloads look correct.
4. Delete or tag the record if you want to keep production analytics clean.

## When to escalate to phone testing

Move to the **Phone Call** tab when any of the following are true:

- You changed audio/voice parameters and need to hear the PSTN leg
- You added AMD, transfers, or webhook actions that only fire on real calls
- You saw latency issues that could be browser-specific and want to confirm the carrier path

Head to [/test/phone-testing](/test/phone-testing) for the full checklist once the web session behaves as expected.
