## How to use this guide

Pick one scenario from each section below every time you ship a change. The scenarios mirror the way our stack is built: web simulation for rapid loops, real phone calls for the PSTN path, and targeted edge cases that exercise context, actions, and clean-up logic.

## 1. Web call smoke tests

<AccordionGroup>
  <Accordion title="Greeting + instructions" icon="hand-wave">
    - Start a fresh web call from the Test Agent drawer
    - Let the agent speak first, then interrupt midway
    - Confirm greeting timing, voice, and the initial instruction block
  </Accordion>
  <Accordion title="Prompt change regression" icon="wand-magic-sparkles">
    - Ask two questions the latest prompt edits were meant to solve
    - Confirm the agent now answers correctly and nothing else regressed
  </Accordion>
  <Accordion title="Context API injection" icon="id-card">
    - Provide contact + agent numbers so the drawer sends a `testContext`
    - Verify the API-derived metadata is mentioned or used in the flow
  </Accordion>
</AccordionGroup>

## 2. Phone call validation

<AccordionGroup>
  <Accordion title="Baseline PSTN run" icon="phone">
    - Place a call using the same From number customers see
    - Capture latency observations and compare to the last release
  </Accordion>
  <Accordion title="AMD + voicemail" icon="voicemail">
    - Switch AMD to **ML**, send the call to voicemail, and confirm the agent exits gracefully without leaving noise
  </Accordion>
  <Accordion title="Action under load" icon="bolt">
    - Trigger at least one booking, transfer, or webhook while on a real call
    - Check Conversations afterwards to confirm payloads were delivered
  </Accordion>
</AccordionGroup>

## 3. Integration & data checks

<CardGroup cols={3}>
  <Card title="Knowledge sources" icon="book">
    Ask one question per attached knowledge base. Confirm the answer references the right source or gracefully declines.
  </Card>
  <Card title="Dynamic instructions" icon="gears">
    Change a test variable value, re-run the web call, and make sure the agent acknowledges the new state.
  </Card>
  <Card title="Campaign prep" icon="calendar">
    If the agent will be used in an outbound campaign, run an outbound web call followed by a phone call to mimic the exact flow campaigns will use.
  </Card>
</CardGroup>

## 4. Failure handling

<AccordionGroup>
  <Accordion title="Silence + timeout" icon="hourglass">
    Stay silent for longer than the configured inactivity window. Confirm the agent follows the escalation/closing message you configured.
  </Accordion>
  <Accordion title="API/tool failure" icon="triangle-exclamation">
    Force an error (e.g., revoke a staging API key) and ensure the agent communicates the failure clearly without exposing stack traces.
  </Accordion>
  <Accordion title="Network drop" icon="plug-circle-xmark">
    Hang up mid-sentence or close the browser to verify wrap-up logic still runs and the conversation record completes.
  </Accordion>
</AccordionGroup>

## 5. Sign-off checklist

Use this quick list to capture results in your ticket or PR description:

- [ ] Web call smoke test passed (notes + timestamp)
- [ ] Phone call validation passed (carrier + device noted)
- [ ] Critical integrations verified (knowledge, APIs, transfers)
- [ ] Failure scenario tested with expected messaging
- [ ] Conversation log reviewed and archived if necessary

Documenting each run means we can reproduce issues quickly and build a regression history for future releases.
