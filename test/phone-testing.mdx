## When to run phone tests

Web calls are perfect for fast iteration, but they bypass carriers. Run at least one real phone call before every release so you can hear codec changes, latency, AMD behaviour, and transfer flows through the exact telephony stack we ship to production.

## Launch a phone test

<Steps>
  <Step title="Open the Test Agent drawer">
    From the agent header, click **Test Agent**.
  </Step>
  <Step title="Switch to Phone Call">
    Use the segmented control to select the **Phone Call** tab.
  </Step>
  <Step title="Pick a From number">
    Choose one of your imported numbers. The combobox uses the same list shown under Telephony → Numbers, including friendly labels so you know what the caller ID will be.
  </Step>
  <Step title="Enter the destination">
    Add the contact's phone number in E.164 format so the call reaches the right person.
  </Step>
  <Step title="Tune AMD">
    Select **Text** for fast keyword-based detection or **ML** for higher accuracy when you expect voicemail.
  </Step>
  <Step title="Start the call">
    Click **Start Test**. The floating widget indicates the dial/outbound state until the callee picks up.
  </Step>
</Steps>

## What to verify during the call

<CardGroup cols={2}>
  <Card title="Audio path" icon="waveform">
    Listen for clipping, low volume, or echo while switching between handset and speakerphone.
  </Card>
  <Card title="Latency" icon="clock">
    Count the beats between speaking and hearing the agent. Flag anything over ~600 ms.
  </Card>
  <Card title="AMD" icon="robot">
    Let voicemail answer to check whether the selected AMD mode behaves correctly.
  </Card>
  <Card title="Actions & transfers" icon="bolt">
    Trigger custom actions, bookings, or transfers to confirm they still fire when audio flows through PSTN.
  </Card>
  <Card title="Context accuracy" icon="id-card">
    Confirm caller-specific prompts or knowledge lookups still work when the call is initiated with real metadata.
  </Card>
  <Card title="End-of-call handling" icon="flag-checkered">
    Ensure wrap-up messaging, automations, and transcripts finalize even if the caller hangs up first.
  </Card>
</CardGroup>

## Scenario checklists

<AccordionGroup>
  <Accordion title="Baseline speaker test" icon="headphones">
    Call from a quiet room on speakerphone, then on a headset. Note volume differences, and verify barging/interruption feels natural.
  </Accordion>
  <Accordion title="Mobile network mix" icon="tower-cell">
    Place calls from different carriers (if available) to surface jitter or codec quirks specific to a network.
  </Accordion>
  <Accordion title="Voicemail/AMD run" icon="voicemail">
    Let the call go to voicemail, capture how AMD classifies it, and ensure the agent does not leave a garbled message.
  </Accordion>
  <Accordion title="Long call stability" icon="hourglass">
    Stay on the line for 5+ minutes to confirm context retention and absence of drift or disconnects.
  </Accordion>
</AccordionGroup>

## After the call

1. Open **Conversations → Recent**.
2. Filter by agent and look for the `phone` badge in the call list.
3. Verify transcript accuracy, goal completion, and any downstream webhooks.
4. Document issues in your ticket before running another iteration.

## Troubleshooting quick reference

- The call never connects → double-check the selected From number is assigned and active.
- The call fails immediately → ensure the `django` API has the right SIP credentials loaded; redeploy if credentials changed.
- No transcript shows up → inspect the calling-service logs for that call ID and confirm the process was running when you placed the call.
- Audio sounds different from the web test → capture the timestamp and carrier so we can reproduce with the same setup.

When the phone pass is clean, you are ready to follow the launch checklist.
