## Why testing matters

Every release touches multiple services: the customer-facing dashboard, the backend API, and the real-time calling layer that drives live conversations. A solid test pass exercises all three so we catch prompt issues, regressions in API contracts, and call quality problems before they reach customers.

## Recommended workflow

<Steps>
  <Step title="Prepare the agent">
    Fill required test variables, confirm the correct knowledge bases are attached, and make a small scripted scenario for what you expect to hear.
  </Step>
  <Step title="Run a web call">
    Use the Test Agent drawer (Web Call tab) for fast iteration. Confirm greetings, instructions, and dynamic context responses.
  </Step>
  <Step title="Trigger a phone call">
    Switch to the Phone Call tab once the web run is clean. Call from the number your customers will see and listen for latency, audio artifacts, or AMD issues.
  </Step>
  <Step title="Review the log">
    Open Conversations → Recent, confirm the transcript, goals, and post-call automations look correct, and jot down any follow-up fixes.
  </Step>
  <Step title="Run automated tests">
    Before opening a pull request, execute the stack-specific test commands listed below so CI sees a clean run.
  </Step>
</Steps>

## Automated test suites

Manual QA only sticks if the automated coverage stays green. Before sharing results:

- **Frontend UI** – run the full UI script your engineering team provides so type-checking, linting, unit tests, and production builds all pass.
- **Backend API** – run the containerized backend test suite to verify endpoints and business logic before releasing new prompt or action behavior.
- **Real-time calling layer** – run the unit + integration suite with coverage enabled to ensure signaling, audio, and action dispatching continue to work at scale.

<Note>Use the internal engineering scripts or CI pipelines for these suites; we intentionally omit repository-specific commands in public documentation.</Note>

## Manual QA exit criteria

- ✅ Web and phone calls match the scripted scenarios (greetings, instructions, fallbacks)
- ✅ Conversation log shows the right metadata, actions, and post-call workflows
- ✅ Relevant automated suites pass locally for the service you touched
- ✅ Outstanding issues are documented in the ticket before handing off to launch or production monitoring
