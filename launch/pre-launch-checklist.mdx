## Why Pre-Launch Validation Matters

Launching an untested agent can lead to poor customer experiences, wasted calls, and support escalations. This checklist ensures every component—from prompts to telephony—has been validated before you route live traffic.

## Agent Configuration Review

<Steps>
  <Step title="Core identity & instructions">
    - Agent name and description are customer-facing appropriate
    - Instructions clearly define scope, tone, and behavior
    - Greeting message is tested for clarity and pronunciation
    - Variables and dynamic context placeholders are populated correctly
  </Step>

  <Step title="Voice pipeline settings">
    - AI model selected (GPT-4, Claude, etc.) and tested for latency
    - Transcriber configured (Deepgram, Azure) with correct language
    - Voice selected and tested for naturalness and speed
    - Custom pronunciations added for technical terms or brand names
    - Ambient sound configured if needed for call center environments
  </Step>

  <Step title="Knowledge base assignment">
    - Relevant knowledge bases are linked to the agent
    - Knowledge content is up-to-date and tested via retrieval queries
    - RAG settings optimized (chunk size, retrieval count)
  </Step>

  <Step title="Actions & integrations">
    - All actions (transfer, booking, custom APIs) configured with correct endpoints
    - API credentials and webhooks tested in Test mode
    - DTMF controls enabled if needed
    - MCP servers connected and functional (if applicable)
  </Step>
</Steps>

## Testing Completion Requirements

Before launch, you must complete both web and phone testing. Use the **Test Overview** checklist:

<CardGroup cols={2}>
  <Card title="Web simulator tests" icon="browser">
    - Run at least 3 complete conversations covering different scenarios
    - Test all configured actions (transfers, bookings, API calls)
    - Verify goal tracking and conversation analytics capture correctly
    - Confirm interruption handling and barge-in behavior
  </Card>

  <Card title="Phone test calls" icon="phone">
    - Complete at least 2 PSTN test calls (headset + speaker)
    - Verify audio quality, echo cancellation, and latency
    - Test DTMF input if configured
    - Confirm voicemail detection (AMD) if using outbound campaigns
  </Card>

  <Card title="Edge case validation" icon="triangle-exclamation">
    - Interrupt agent mid-sentence to test barge-in
    - Stay silent to trigger inactivity handling
    - Request out-of-scope information to test guardrails
    - Test multilingual support if enabled
  </Card>

  <Card title="Log review" icon="file-magnifying-glass">
    - Open test conversations in **Conversations → Logs**
    - Verify transcripts are accurate and complete
    - Check goal outcomes and custom data payloads
    - Confirm post-call webhooks fired correctly
  </Card>
</CardGroup>

## Telephony Setup Verification

<Steps>
  <Step title="Phone numbers or SIP trunks">
    Navigate to **Telephony → Numbers** and confirm:
    - At least one phone number is provisioned and active
    - If using your own carrier, SIP credentials are configured in `backend/telephony`
    - Test inbound routing by calling the number and verifying agent answers
  </Step>

  <Step title="Concurrency limits">
    Check your account's concurrent call capacity:
    - Review plan limits in **Accounts → Usage**
    - Configure overflow or queueing if expecting high volume
    - Set business hours to control when agent is available
  </Step>

  <Step title="Carrier quality check">
    Run a live test call and monitor logs for:
    - SIP registration success
    - Audio codec negotiation (Opus, PCMU)
    - Latency measurements (target < 300ms RTT)
  </Step>
</Steps>

## Security & Access Control

<Accordion title="API key management">
  - Generate production API keys in **Accounts → API Keys**
  - Store keys securely (environment variables, secrets manager)
  - Limit key permissions to minimum required scope
  - Never commit keys to version control
</Accordion>

<Accordion title="Team access">
  - Ensure correct team members have deployment permissions
  - Set up role-based access control for production agents
  - Document who has authority to modify live configurations
</Accordion>

<Accordion title="Webhook security">
  - Configure webhook signature verification for post-call events
  - Use HTTPS endpoints only
  - Test webhook delivery and retry logic
</Accordion>

## Monitoring & Alerting Setup

Before launch, configure observability:

- **Real-time monitoring**: Open **Conversations → Live Monitor** to watch active calls
- **Alert thresholds**: Set up alerts for error rates, latency spikes, or capacity limits
- **Dashboard access**: Bookmark analytics dashboard for quick health checks
- **Support integration**: Connect logs to your support tool (Slack, PagerDuty, etc.)

## Rollback Plan

Prepare for worst-case scenarios:

1. **Backup agent version**: Keep a tested fallback agent configuration
2. **Manual override**: Document how to quickly reassign phone numbers to human queues
3. **Emergency contacts**: List who can execute rollback procedures
4. **Communication plan**: Draft customer notification templates if issues occur

## Final Pre-Launch Checklist

Before clicking deploy:

- <Icon icon="square" /> Agent tested via web simulator (3+ scenarios)
- <Icon icon="square" /> Phone test calls completed (2+ calls)
- <Icon icon="square" /> All actions and integrations verified in logs
- <Icon icon="square" /> Knowledge base retrieval tested with real queries
- <Icon icon="square" /> Phone number or SIP trunk assigned and tested
- <Icon icon="square" /> Concurrency limits understood and configured
- <Icon icon="square" /> Production API keys generated and secured
- <Icon icon="square" /> Monitoring dashboard and alerts configured
- <Icon icon="square" /> Rollback plan documented and team briefed
- <Icon icon="square" /> Customer support team trained on agent capabilities
