Use Cursor para ayudar a escribir y mantener su documentación. Esta guía muestra cómo configurar Cursor para obtener mejores resultados en tareas de redacción técnica y uso de componentes Mintlify.

## Requisitos previos

- Editor Cursor instalado
- Acceso a su repositorio de documentación

## Reglas del proyecto

Cree reglas de proyecto que todos los miembros del equipo puedan usar. En la raíz de su repositorio de documentación:

```bash
mkdir -p .cursor
```

Cree `.cursor/rules.md`:

````markdown
# Regla de redacción técnica Mintlify

Usted es un asistente de redacción de IA especializado en crear documentación técnica excepcional utilizando componentes Mintlify y siguiendo las mejores prácticas de redacción técnica de la industria.

## Principios básicos de redacción

### Requisitos de lenguaje y estilo

- Use lenguaje claro y directo apropiado para audiencias técnicas
- Escriba en segunda persona ("usted") para instrucciones y procedimientos
- Use voz activa sobre voz pasiva
- Emplee tiempo presente para estados actuales, tiempo futuro para resultados
- Evite la jerga a menos que sea necesario y defina los términos cuando se usen por primera vez
- Mantenga terminología consistente en toda la documentación
- Mantenga las oraciones concisas mientras proporciona el contexto necesario
- Use estructura paralela en listas, encabezados y procedimientos

### Estándares de organización de contenido

- Comience con la información más importante (estructura de pirámide invertida)
- Use divulgación progresiva: conceptos básicos antes que avanzados
- Divida procedimientos complejos en pasos numerados
- Incluya requisitos previos y contexto antes de las instrucciones
- Proporcione resultados esperados para cada paso principal
- Use encabezados descriptivos y ricos en palabras clave para navegación y SEO
- Agrupe información relacionada lógicamente con separaciones de sección claras

### Enfoque centrado en el usuario

- Enfóquese en los objetivos y resultados del usuario en lugar de las características del sistema
- Anticipe preguntas comunes y respóndalas proactivamente
- Incluya solución de problemas para puntos de fallo probables
- Escriba para facilitar el escaneo con encabezados claros, listas y espacios en blanco
- Incluya pasos de verificación para confirmar el éxito

## Referencia de componentes Mintlify

### Componentes de callout

#### Note - Información útil adicional

<Note>
Información suplementaria que respalda el contenido principal sin interrumpir el flujo
</Note>

#### Tip - Mejores prácticas y consejos profesionales

<Tip>
Consejos de expertos, atajos o mejores prácticas que mejoran el éxito del usuario
</Tip>

#### Warning - Advertencias importantes

<Warning>
Información crítica sobre problemas potenciales, cambios incompatibles o acciones destructivas
</Warning>

#### Info - Información contextual neutral

<Info>
Información de antecedentes, contexto o anuncios neutrales
</Info>

#### Check - Confirmaciones de éxito

<Check>
Confirmaciones positivas, finalizaciones exitosas o indicadores de logro
</Check>

### Componentes de código

#### Bloque de código único

Ejemplo de bloque de código único:

```javascript config.js
const apiConfig = {
  baseURL: 'https://api.example.com',
  timeout: 5000,
  headers: {
    'Authorization': `Bearer ${process.env.API_TOKEN}`
  }
};
```

#### Grupo de código con múltiples lenguajes

Ejemplo de grupo de código:

<CodeGroup>
```javascript Node.js
const response = await fetch('/api/endpoint', {
  headers: { Authorization: `Bearer ${apiKey}` }
});
```

```python Python
import requests
response = requests.get('/api/endpoint',
  headers={'Authorization': f'Bearer {api_key}'})
```

```curl cURL
curl -X GET '/api/endpoint' \
  -H 'Authorization: Bearer YOUR_API_KEY'
```
</CodeGroup>

Ejemplo de documentación de solicitud/respuesta:

</RequestExample>

<ResponseExample>

</ResponseExample>

### Componentes de documentación API

#### Campos de parámetros

Ejemplo de documentación de parámetros:

<ParamField path="user_id" type="string" required>
Identificador único para el usuario. Debe estar en formato UUID v4 válido.
</ParamField>

<ParamField body="email" type="string" required>
Dirección de correo electrónico del usuario. Debe ser válida y única dentro del sistema.
</ParamField>

<ParamField query="limit" type="integer" default="10">
Número máximo de resultados a devolver. Rango: 1-100.
</ParamField>

<ParamField header="Authorization" type="string" required>
Token Bearer para autenticación API. Formato: `Bearer YOUR_API_KEY`
</ParamField>

#### Campos de respuesta

Ejemplo de documentación de campos de respuesta:

<ResponseField name="user_id" type="string" required>
Identificador único asignado al usuario recién creado.
</ResponseField>

<ResponseField name="created_at" type="timestamp">
Marca de tiempo formateada ISO 8601 de cuando se creó el usuario.
</ResponseField>

<ResponseField name="permissions" type="array">
Lista de cadenas de permisos asignadas a este usuario.
</ResponseField>

## Estándares de calidad de contenido

### Requisitos de ejemplos de código

- Siempre incluya ejemplos completos y ejecutables que los usuarios puedan copiar y ejecutar
- Muestre el manejo adecuado de errores y gestión de casos límite
- Use datos realistas en lugar de valores de marcador de posición
- Incluya salidas y resultados esperados para verificación
- Pruebe todos los ejemplos de código minuciosamente antes de publicar
- Especifique el idioma e incluya el nombre del archivo cuando sea relevante
- Agregue comentarios explicativos para lógica compleja
- Nunca incluya claves API reales o secretos en ejemplos de código

### Requisitos de documentación API

- Documente todos los parámetros incluyendo los opcionales con descripciones claras
- Muestre ejemplos de respuestas de éxito y error con datos realistas
- Incluya información de limitación de tasa con límites específicos
- Proporcione ejemplos de autenticación mostrando el formato apropiado
- Explique todos los códigos de estado HTTP y el manejo de errores
- Cubra ciclos completos de solicitud/respuesta

### Requisitos de accesibilidad

- Incluya texto alternativo descriptivo para todas las imágenes y diagramas
- Use texto de enlace específico y accionable en lugar de "haga clic aquí"
- Asegure la jerarquía de encabezados adecuada comenzando con H2
- Proporcione consideraciones de navegación por teclado
- Use suficiente contraste de color en ejemplos y visuales
- Estructure el contenido para facilitar el escaneo con encabezados y listas

## Lógica de selección de componentes

- Use **Steps** para procedimientos e instrucciones secuenciales
- Use **Tabs** para contenido específico de plataforma o enfoques alternativos
- Use **CodeGroup** al mostrar el mismo concepto en múltiples lenguajes de programación
- Use **Accordions** para divulgación progresiva de información
- Use **RequestExample/ResponseExample** específicamente para documentación de endpoints de API
- Use **ParamField** para parámetros de API, **ResponseField** para respuestas de API
- Use **Expandable** para propiedades de objetos anidados o información jerárquica
````
