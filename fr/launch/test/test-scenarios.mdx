## Pourquoi les scénarios de test sont importants

Des scénarios de test répétables garantissent une qualité cohérente à travers les mises à jour de l'agent. Créez une suite de tests qui couvre :
- **Parcours optimaux** - Interactions clients idéales
- **Cas limites** - Scénarios inhabituels mais valides
- **Conditions d'erreur** - Comment l'agent gère les échecs
- **Tests de régression** - Empêcher de casser les fonctionnalités existantes

## Scénarios de test principaux

### 1. Accueil et introduction

**Scénario** : Le client appelle pour la première fois, a besoin de comprendre avec qui il parle.

<Accordion title="Étapes de test">
  1. Démarrer l'appel (test web ou téléphonique)
  2. Attendre l'accueil de l'agent
  3. Ne pas parler immédiatement (tester si l'agent sollicite)
  4. Demander : "À qui je parle ?"
  5. Vérifier que l'agent se présente clairement

  **Résultats attendus** :
  - L'accueil se joue dans les 2 secondes du début de l'appel
  - L'agent indique clairement le nom/l'entreprise
  - L'agent propose de l'aide de manière proactive
  - Ton professionnel et amical
</Accordion>

<Accordion title="Variations">
  - Appeler pendant les heures d'ouverture vs après les heures
  - Appeler depuis différents numéros de téléphone (tester la reconnaissance de l'ID appelant)
  - Appeler avec différentes langues activées (test multilingue)
</Accordion>

---

### 2. Recherche d'informations (base de connaissances)

**Scénario** : Le client pose une question dont la réponse se trouve dans la base de connaissances.

<Accordion title="Étapes de test">
  1. Poser une question clairement répondue dans la KB (ex : "Quels sont vos horaires d'ouverture ?")
  2. Vérifier que l'agent récupère les bonnes informations
  3. Poser une question de suivi
  4. Vérifier que l'agent maintient le contexte

  **Résultats attendus** :
  - L'agent répond avec précision depuis le contenu KB
  - La réponse est concise (ne lit pas l'article KB entier)
  - L'agent propose une aide connexe
  - Les questions de suivi sont gérées correctement
</Accordion>

<Accordion title="Cas limites">
  - Poser une question avec une formulation légèrement différente du contenu KB
  - Poser une question combinant plusieurs sujets KB
  - Poser une question avec une intention ambiguë
  - Poser une question NON dans la KB (l'agent devrait dire "Je ne sais pas")
</Accordion>

---

### 3. Exécution d'action (appel API)

**Scénario** : Le client demande une action nécessitant un appel API externe.

<Accordion title="Étapes de test">
  1. Demander une action (ex : "Vérifiez l'état de ma commande 12345")
  2. Vérifier que l'agent confirme la compréhension
  3. Attendre que l'appel API se termine
  4. Vérifier que l'agent communique le résultat correctement

  **Résultats attendus** :
  - L'agent extrait les paramètres requis (numéro de commande)
  - L'agent appelle le bon endpoint API
  - L'agent attend la réponse sans erreur de timeout
  - L'agent communique le résultat naturellement
  - Si l'API échoue, l'agent gère avec élégance
</Accordion>

<Accordion title="Scénarios d'échec">
  - L'API retourne une erreur (500, 404)
  - L'API expire (réponse lente)
  - L'API retourne un format inattendu
  - Paramètre requis manquant (l'agent devrait le demander)
</Accordion>

---

### 4. Transfert d'appel

**Scénario** : Le client a besoin de parler avec un agent humain.

<Accordion title="Étapes de test">
  1. Demander un transfert (ex : "J'ai besoin de parler à quelqu'un")
  2. Vérifier que l'agent confirme le transfert
  3. Vérifier que l'agent annonce qui/quel département
  4. (Test téléphonique uniquement) Vérifier que l'appel est effectivement transféré

  **Résultats attendus** :
  - L'agent reconnaît immédiatement la demande
  - L'agent explique où vous serez transféré
  - L'agent tente le transfert
  - Le transfert se termine avec succès (test téléphonique)
</Accordion>

<Accordion title="Cas limites">
  - Demander un transfert vers un département spécifique
  - Demander un transfert en dehors des heures d'ouverture
  - Demander un transfert après une action échouée
  - Plusieurs demandes de transfert dans le même appel
</Accordion>

---

### 5. Gestion des interruptions

**Scénario** : Le client interrompt l'agent en milieu de phrase.

<Accordion title="Étapes de test">
  1. Poser une question qui provoque une longue réponse
  2. Pendant que l'agent parle, interrompre avec une nouvelle question
  3. Vérifier que l'agent arrête de parler et écoute
  4. Vérifier que l'agent traite la nouvelle question

  **Résultats attendus** :
  - L'agent arrête de parler immédiatement (&lt;500ms)
  - L'agent traite la nouvelle entrée
  - L'agent ne reprend pas la réponse précédente
  - La conversation se déroule naturellement après l'interruption
</Accordion>

<Accordion title="Tests de stress">
  - Interrompre plusieurs fois rapidement
  - Interrompre avec une question non liée
  - Interrompre avec juste du bruit (pas de mots)
  - Rester silencieux après avoir interrompu
</Accordion>

---

### 6. Inactivité / silence

**Scénario** : Le client arrête de répondre pendant la conversation.

<Accordion title="Étapes de test">
  1. Démarrer la conversation normalement
  2. L'agent pose une question
  3. Rester complètement silencieux (ne pas répondre)
  4. Attendre 10-15 secondes
  5. Vérifier que l'agent sollicite : "Êtes-vous toujours là ?"
  6. Rester silencieux à nouveau
  7. Vérifier que l'agent termine finalement l'appel ou transfère

  **Résultats attendus** :
  - Première sollicitation après ~10 secondes
  - Deuxième sollicitation si toujours silencieux
  - Après 2-3 sollicitations, l'agent termine l'appel avec élégance
  - Journalisation appropriée du timeout d'inactivité
</Accordion>

---

### 7. Entrée DTMF (clavier)

**Scénario** : Le client utilise le clavier téléphonique pour l'entrée (si activé).

<Accordion title="Étapes de test">
  1. L'agent sollicite : "Appuyez sur 1 pour les ventes, 2 pour le support"
  2. Appuyer sur une touche du clavier téléphonique
  3. Vérifier que l'agent détecte et répond
  4. Tester toutes les touches : 0-9, *, #

  **Résultats attendus** :
  - Toutes les touches détectées correctement
  - L'agent confirme l'entrée ("Vous avez appuyé sur 1")
  - L'agent route vers la bonne action/département
  - Les touches invalides sont gérées (ex : appuyer sur 9 quand seul 1-2 est valide)
</Accordion>

---

### 8. Support multilingue

**Scénario** : Le client parle dans une langue différente (si multilingue activé).

<Accordion title="Étapes de test">
  1. Démarrer l'appel en anglais
  2. Passer à l'espagnol en milieu d'appel
  3. Vérifier que l'agent détecte la langue et répond en espagnol
  4. Repasser à l'anglais
  5. Vérifier que l'agent s'adapte

  **Résultats attendus** :
  - Détection de la langue en 1-2 phrases
  - L'agent répond dans la langue détectée
  - Transcription précise pour les deux langues
  - Sélection de voix appropriée pour la langue
</Accordion>

---

### 9. Conversation longue (test d'endurance)

**Scénario** : Conversation étendue testant la rétention du contexte.

<Accordion title="Étapes de test">
  1. Avoir une conversation couvrant 10+ tours
  2. Référencer des points de conversation antérieurs
  3. Poser des questions complexes en plusieurs parties
  4. Vérifier que l'agent se souvient du contexte

  **Résultats attendus** :
  - L'agent conserve l'historique de conversation
  - L'agent ne répète pas les informations déjà fournies
  - L'agent ne demande pas des informations déjà fournies par le client
  - La qualité d'appel reste stable pendant 10+ minutes
</Accordion>

---

### 10. Cas adverses / limites

**Scénario** : Scénarios de clients difficiles.

<Accordion title="Étapes de test">
  **Client confus** :
  - Donner des réponses confuses et peu claires
  - Changer de sujet en milieu de conversation
  - Poser des questions non liées

  **Client frustré** :
  - Exprimer de la frustration : "Ça ne marche pas !"
  - Exiger de parler avec un humain
  - Utiliser un langage fort (tester les garde-fous)

  **Client testeur** :
  - Essayer de faire dire des choses inappropriées à l'agent
  - Poser des questions hors sujet
  - Tester les limites des connaissances de l'agent

  **Résultats attendus** :
  - L'agent reste serviable et professionnel
  - L'agent propose de transférer quand approprié
  - L'agent ne s'engage pas avec des demandes inappropriées
  - L'agent dit avec élégance "Je ne peux pas aider avec ça"
</Accordion>

## Créer votre suite de tests

<Steps>
  <Step title="Documenter les scénarios principaux">
    Créez un tableau des scénarios que votre agent doit gérer :

    | Scénario | Priorité | Critères de réussite |
    |----------|----------|---------------|
    | Accueil | P0 (critique) | Présentation claire, &lt;2s |
    | Recherche KB | P0 | Réponse précise, récupère depuis KB |
    | Appel API | P1 (important) | S'exécute avec succès, &lt;5s |
    | Transfert | P1 | Se connecte à la bonne destination |
    | DTMF | P2 (souhaitable) | Détecte toutes les touches correctement |
  </Step>

  <Step title="Créer des scripts de test">
    Écrivez un dialogue exact pour chaque scénario :

    **Exemple de script** :
    ```
    Scénario : Recherche d'état de commande
    Testeur : "Bonjour, quel est l'état de ma commande ?"
    Agent attendu : "Je peux vous aider avec ça. Quel est votre numéro de commande ?"
    Testeur : "12345"
    Agent attendu : "Laissez-moi vérifier... [Appel API] Votre commande a été expédiée et arrivera le 15 janvier."
    Résultat : RÉUSSI/ÉCHOUÉ
    ```
  </Step>

  <Step title="Exécuter la suite complète régulièrement">
    **Quand tester** :
    - Avant chaque déploiement en production
    - Après des changements d'instructions majeurs
    - Hebdomadairement pour les agents critiques
    - Après les mises à jour de la plateforme

    **Suivi** :
    Utilisez une feuille de calcul ou un outil de gestion de tests pour suivre :
    - Date testée
    - Réussi/échoué par scénario
    - Notes sur les échecs
    - Tendances au fil du temps
  </Step>

  <Step title="Automatiser quand possible">
    Pour les scénarios critiques, automatisez les tests :

    ```python
    # Exemple de test automatisé
    def test_greeting():
        call = client.test_calls.create(
            agent_id="agent_123",
            scenario="greeting",
            expected_keywords=["welcome", "help"]
        )

        assert call.duration_seconds < 5
        assert "welcome" in call.transcript.lower()
        assert call.status == "success"
    ```

    Exécutez les tests automatisés selon un calendrier ou avant le déploiement.
  </Step>
</Steps>

## Matrice de couverture des tests

Assurez-vous de tester toutes les capacités de l'agent :

| Capacité | Scénario de test | Statut |
|------------|---------------|--------|
| **Conversation de base** |
| Accueil | Test d'accueil du premier appel | ✓ |
| Instructions | Test de suivi des règles de l'agent | ✓ |
| Contexte | Conversation multi-tours | ✓ |
| **Connaissances** |
| Récupération KB | Test de question FAQ | ✓ |
| Précision KB | Vérifier les réponses correctes | ✓ |
| Gestion de l'inconnu | Poser une question sans réponse | ✓ |
| **Actions** |
| Appels API | Déclencher chaque action personnalisée | ⚠️ En cours |
| Transferts | Tester chaque destination de transfert | ⚠️ En cours |
| Réservation | Compléter une réservation de calendrier | ⏳ Pas commencé |
| **Fonctionnalités vocales** |
| Interruptions | Test de barge-in | ✓ |
| Silence | Test de timeout d'inactivité | ✓ |
| DTMF | Test d'entrée clavier | N/A |
| **Qualité** |
| Audio | Métriques de qualité d'appel | ✓ |
| Latence | Temps de réponse &lt;2s | ✓ |
| Précision | WER transcription &lt;10% | ✓ |
