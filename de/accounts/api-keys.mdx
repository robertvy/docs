## Übersicht

API-Schlüssel ermöglichen programmatischen Zugriff auf die itellicoAI-Plattform und erlauben es Ihnen, Agenten in Ihre Anwendungen zu integrieren, Aufgaben zu automatisieren und benutzerdefinierte Workflows zu erstellen.

## Was sind API-Schlüssel?

API-Schlüssel sind sichere Tokens, die Ihre API-Anfragen authentifizieren, ohne dass Benutzer-Anmeldedaten erforderlich sind.

**Hauptmerkmale:**
- **Account-gebunden**: Jeder Schlüssel gehört zu einem bestimmten Konto
- **Geheim**: Behandeln Sie ihn wie ein Passwort - niemals teilen oder in Versionskontrolle einchecken
- **Widerrufbar**: Kann jederzeit deaktiviert oder gelöscht werden
- **Nachverfolgbar**: Überwachen Sie den Zeitstempel der letzten Verwendung und Aktivität

## API-Schlüssel verwenden

### Authentifizierungs-Header

Fügen Sie Ihren API-Schlüssel in den `Authorization`-Header ein:

```bash
curl https://api.itellico.ai/v1/agents \
  -H "Authorization: Bearer sk-a1b2c3d4.xyz789..." \
  -H "Content-Type: application/json"
```

### Account-Kontext

API-Schlüssel sind auf ihr Konto beschränkt. Wenn Sie einen Schlüssel in einem Elternkonto erstellen, kann er auf das Elternkonto und alle Unterkonten zugreifen.

Um auf ein bestimmtes Konto zuzugreifen, fügen Sie die Account-ID in den URL-Pfad ein:

```bash
# Auf Elternkonto zugreifen
curl https://api.itellico.ai/v1/accounts/me/agents \
  -H "Authorization: Bearer sk-a1b2c3d4.xyz789..." \
  -H "Content-Type: application/json"

# Auf bestimmtes Unterkonto zugreifen
curl https://api.itellico.ai/v1/accounts/{account-id}/agents \
  -H "Authorization: Bearer sk-a1b2c3d4.xyz789..." \
  -H "Content-Type: application/json"
```

### SDKs

Verwendung unserer offiziellen SDKs:

<CodeGroup>

```python Python
from itellico import ItellicoAI

client = ItellicoAI(api_key="sk-a1b2c3d4.xyz789...")

# Agenten auflisten
agents = client.agents.list()
```

```typescript TypeScript
import { ItellicoAI } from '@itellico/sdk';

const client = new ItellicoAI({
  apiKey: 'sk-a1b2c3d4.xyz789...'
});

// Agenten auflisten
const agents = await client.agents.list();
```

</CodeGroup>

## API-Schlüssel verwalten

### Schlüssel anzeigen

Die API-Schlüssel-Seite zeigt:
- **Bezeichnung**: Ihr beschreibender Name
- **Teilschlüssel**: Erste Zeichen (z.B. `sk-a1b2c3d4...`)
- **Status**: Aktiv, Widerrufen oder Abgelaufen
- **Erstellt**: Wann der Schlüssel erstellt wurde
- **Zuletzt verwendet**: Wann er zuletzt für eine API-Anfrage verwendet wurde
- **Läuft ab**: Ablaufdatum (falls festgelegt)

<Note>
Der vollständige Schlüssel wird nach der Erstellung **niemals** angezeigt - nur die ersten Zeichen zur Identifikation.
</Note>

### Schlüssel bearbeiten

Sie können aktualisieren:
- **Bezeichnung**: Ändern Sie den beschreibenden Namen
- **Ablaufdatum**: Verlängern oder Ablaufdatum festlegen

Sie **können nicht** ändern:
- Den Schlüssel selbst (erstellen Sie stattdessen einen neuen)
- Das Konto, zu dem er gehört

### Schlüssel widerrufen

Um einen Schlüssel vorübergehend zu deaktivieren, ohne ihn zu löschen:

1. Gehen Sie zu **Einstellungen → API-Schlüssel**
2. Finden Sie den Schlüssel in der Liste
3. Klicken Sie auf **Widerrufen**
4. Der Schlüssel wird sofort deaktiviert

**Warum widerrufen statt löschen?**
- Behält den Schlüssel in Logs zur Prüfung
- Kann reaktiviert werden, falls versehentlich widerrufen
- Bewahrt Erstellungsdatum und Verlauf

### Schlüssel reaktivieren

Um einen widerrufenen Schlüssel wiederherzustellen:

1. Finden Sie den widerrufenen Schlüssel in der Liste
2. Klicken Sie auf **Reaktivieren**
3. Der Schlüssel funktioniert sofort

### Schlüssel löschen

Um einen Schlüssel dauerhaft zu entfernen:

1. Gehen Sie zu **Einstellungen → API-Schlüssel**
2. Finden Sie den Schlüssel in der Liste
3. Klicken Sie auf das Menü (⋯) → **Löschen**
4. Löschen bestätigen

<Warning>
Die Löschung ist dauerhaft und kann nicht rückgängig gemacht werden. Der Schlüssel funktioniert sofort nicht mehr.
</Warning>

## Schlüsselstatus

| Status | Beschreibung | API-Zugriff |
|--------|-------------|------------|
| **Aktiv** | Schlüssel funktioniert normal | ✅ Ja |
| **Widerrufen** | Manuell deaktiviert | ❌ Nein |
| **Abgelaufen** | Nach Ablaufdatum | ❌ Nein |

## Best Practices

<AccordionGroup>
  <Accordion title="Ein Schlüssel pro Umgebung" icon="server">
    Erstellen Sie separate Schlüssel für jede Umgebung:

    - **Entwicklung**: "Dev-Server-Schlüssel"
    - **Staging**: "Staging-Umgebung"
    - **Produktion**: "Produktionsserver"

    **Vorteile:**
    - Dev-Schlüssel widerrufen, ohne Produktion zu beeinträchtigen
    - Nutzung nach Umgebung nachverfolgen
    - Unterschiedliche Ablaufrichtlinien pro Umgebung
  </Accordion>

  <Accordion title="Umgebungsvariablen verwenden" icon="code">
    Codieren Sie API-Schlüssel niemals fest in Ihren Quellcode.

    **Gut:**
    ```python
    import os
    api_key = os.environ['ITELLICO_API_KEY']
    ```

    **Schlecht:**
    ```python
    api_key = "sk-a1b2c3d4.xyz789..."  # MACHEN SIE DAS NICHT!
    ```

    **Umgebungsdateien (.env):**
    ```bash
    ITELLICO_API_KEY=sk-a1b2c3d4.xyz789...
    ```

    **Zu .gitignore hinzufügen:**
    ```
    .env
    .env.local
    *.key
    ```
  </Accordion>

  <Accordion title="Regelmäßige Schlüsselrotation" icon="rotate">
    Rotieren Sie API-Schlüssel regelmäßig aus Sicherheitsgründen:

    **Empfohlener Zeitplan:**
    - **Produktion**: Alle 90 Tage
    - **Staging**: Alle 180 Tage
    - **Entwicklung**: Jährlich oder bei Entwicklerwechsel

    **Rotationsprozess:**
    1. Neuen Schlüssel mit Ablaufdatum erstellen
    2. Anwendungen mit neuem Schlüssel aktualisieren
    3. Gründlich testen
    4. Alten Schlüssel widerrufen
    5. Alten Schlüssel nach 30 Tagen löschen
  </Accordion>

  <Accordion title="Ablaufdaten festlegen" icon="calendar-xmark">
    Verwenden Sie Ablaufdaten für:
    - **Temporärer Zugriff**: Auftragnehmer, Demos, POCs
    - **Erzwungene Rotation**: Produktionsschlüssel auf 90 Tage Ablauf gesetzt
    - **Zeitlich begrenzte Funktionen**: Beta-Tests, Testversionen

    **Beispiel:**
    - Schlüssel am 1. Januar erstellen
    - Ablauf auf 1. April setzen (90 Tage)
    - 7 Tage vor Ablauf benachrichtigt werden
    - Schlüssel vor Ablauf rotieren
  </Accordion>

  <Accordion title="Nutzung überwachen" icon="chart-line">
    Überprüfen Sie regelmäßig die Zeitstempel "Zuletzt verwendet":

    - **Nie verwendet**: Ungenutzte Schlüssel löschen
    - **Alter Zeitstempel**: Könnte sicher widerrufen werden
    - **Aktuelle Aktivität**: Schlüssel ist aktiv

    Überprüfen Sie Ihre API-Schlüssel monatlich, um inaktive zu entfernen.
  </Accordion>

  <Accordion title="Beschreibende Bezeichnungen" icon="tag">
    Verwenden Sie klare, beschreibende Schlüsselnamen:

    **Gute Bezeichnungen:**
    - "Produktions-API-Server - us-east-1"
    - "Mobile App - iOS - Produktion"
    - "CI/CD - GitHub Actions"
    - "Webhook-Handler - Staging"

    **Schlechte Bezeichnungen:**
    - "API-Schlüssel 1"
    - "Test"
    - "Mein Schlüssel"
    - "Neuer Schlüssel"
  </Accordion>

  <Accordion title="Secrets-Management" icon="vault">
    Verwenden Sie einen Secrets Manager in der Produktion:

    **Optionen:**
    - AWS Secrets Manager
    - HashiCorp Vault
    - Azure Key Vault
    - Google Secret Manager
    - 1Password / LastPass (für Teams)

    **Vorteile:**
    - Zentralisierte Secret-Speicherung
    - Automatische Rotation
    - Audit-Logs
    - Zugriffskontrollen
  </Accordion>
</AccordionGroup>

## Sicherheitsüberlegungen

<Warning>
**Checken Sie niemals API-Schlüssel in Versionskontrolle ein!**

Wenn Sie versehentlich einen Schlüssel einchecken:
1. Widerrufen Sie den Schlüssel sofort
2. Erstellen Sie einen neuen Schlüssel
3. Verwenden Sie `git filter-branch` oder BFG Repo-Cleaner, um ihn aus dem Verlauf zu entfernen
4. Force Push zum Remote
5. Benachrichtigen Sie Teammitglieder, die neueste Version zu pullen
</Warning>

### Wenn ein Schlüssel kompromittiert ist

<Steps>
  <Step title="Sofort widerrufen">
    Gehen Sie zu **Einstellungen → API-Schlüssel** und widerrufen Sie den kompromittierten Schlüssel
  </Step>

  <Step title="Neuen Schlüssel erstellen">
    Generieren Sie einen Ersatzschlüssel mit neuer Bezeichnung
  </Step>

  <Step title="Anwendungen aktualisieren">
    Deployen Sie den neuen Schlüssel in allen betroffenen Anwendungen
  </Step>

  <Step title="Logs überprüfen">
    Überprüfen Sie Nutzungslogs auf verdächtige Aktivitäten
  </Step>

  <Step title="Team benachrichtigen">
    Informieren Sie Ihr Team über den Vorfall
  </Step>

  <Step title="Untersuchen">
    Ermitteln Sie, wie der Schlüssel kompromittiert wurde und verhindern Sie Wiederholung
  </Step>
</Steps>

## Fehlerbehebung

<AccordionGroup>
  <Accordion title="401 Unauthorized" icon="lock">
    **Ursachen:**
    - Ungültiger API-Schlüssel
    - Widerrufener oder abgelaufener Schlüssel
    - Fehlender `Authorization`-Header
    - Falsches Header-Format

    **Lösungen:**
    - Überprüfen Sie, ob der Schlüssel in den Einstellungen aktiv ist
    - Prüfen Sie den Header: `Authorization: Bearer sk-...`
    - Stellen Sie sicher, dass keine zusätzlichen Leerzeichen oder Zeichen vorhanden sind
    - Erstellen Sie einen neuen Schlüssel, falls verloren
  </Accordion>

  <Accordion title="403 Forbidden" icon="ban">
    **Ursachen:**
    - Schlüssel hat keinen Zugriff auf das angeforderte Konto
    - Anforderung eines Unterkontos, auf das der Schlüssel nicht zugreifen kann
    - Konto ist inaktiv

    **Lösungen:**
    - Überprüfen Sie, ob der Schlüssel im Elternkonto oder dem Unterkonto erstellt wurde, auf das Sie zugreifen
    - Prüfen Sie, ob die Account-ID in der URL korrekt ist
    - Überprüfen Sie, ob der Kontostatus aktiv ist
  </Accordion>

  <Accordion title="429 Too Many Requests" icon="gauge-high">
    **Ursachen:**
    - Ratenlimits überschritten
    - Zu viele gleichzeitige Anfragen

    **Lösungen:**
    - Implementieren Sie exponentielles Backoff
    - Cachen Sie Antworten wenn möglich
    - Upgraden Sie auf höheren Tarif
    - Verteilen Sie Anfragen über die Zeit
  </Accordion>

  <Accordion title="Schlüssel funktioniert nach Erstellung nicht" icon="circle-xmark">
    **Mögliche Ursachen:**
    - Falscher Schlüssel verwendet (Präfix vs. vollständiger Schlüssel)
    - Zusätzliche Zeichen (Zeilenumbrüche, Leerzeichen)
    - Sofort nach Erstellung widerrufen

    **Lösungen:**
    - Kopieren Sie den vollständigen Schlüssel einschließlich `sk-`-Präfix
    - Entfernen Sie Leerzeichen aus dem gespeicherten Schlüssel
    - Überprüfen Sie, ob der Status "Aktiv" ist
  </Accordion>
</AccordionGroup>

## FAQs

<AccordionGroup>
  <Accordion title="Wie viele API-Schlüssel kann ich erstellen?">
    Keine harte Grenze, aber wir empfehlen:
    - Kleine Teams: 3-5 Schlüssel
    - Mittlere Teams: 10-15 Schlüssel
    - Enterprise: Nach Bedarf pro Umgebung
  </Accordion>

  <Accordion title="Kann ich einen Schlüssel über mehrere Konten verwenden?">
    API-Schlüssel, die in einem Elternkonto erstellt wurden, können auf das Elternkonto und alle seine Unterkonten zugreifen. Schlüssel, die in einem Unterkonto erstellt wurden, können nur auf dieses Unterkonto zugreifen.
  </Accordion>

  <Accordion title="Was passiert, wenn ich einen Schlüssel lösche?">
    API-Anfragen mit diesem Schlüssel schlagen sofort mit 401 Unauthorized fehl. Aktualisieren Sie zuerst Ihre Anwendungen!
  </Accordion>

  <Accordion title="Kann ich sehen, wofür ein Schlüssel verwendet wurde?">
    Ja. Der Zeitstempel "Zuletzt verwendet" zeigt, wann er zuletzt aufgerufen wurde. Für detaillierte Logs wenden Sie sich an den Support bezüglich API-Zugriffslogs.
  </Accordion>

  <Accordion title="Laufen API-Schlüssel automatisch ab?">
    Nur wenn Sie ein Ablaufdatum festlegen. Andernfalls bleiben Schlüssel aktiv, bis sie widerrufen oder gelöscht werden.
  </Accordion>

  <Accordion title="Können Unterkonten eigene API-Schlüssel haben?">
    Ja! Jedes Unterkonto kann unabhängige API-Schlüssel erstellen, die auf den Kontext dieses Unterkontos beschränkt sind.
  </Accordion>
</AccordionGroup>
