## Warum Telefontests wichtig sind

Während der Web-Simulator die meisten Probleme erkennt, validiert der Telefontest die vollständige End-to-End-Erfahrung einschließlich:
- PSTN-Audioqualität und Latenz
- Echte DTMF-Töne von Telefontastaturen
- Tatsächliche Anrufweiterleitungen und Routing
- Mailbox-Erkennung (AMD)
- Verhalten von Mobil- und Festnetzanrufen

**Empfehlung**: Schließen Sie zunächst die Web-Simulator-Tests ab und validieren Sie dann mit 2-3 Telefontestanrufen vor dem Launch.

## Durchführung eines Telefontests

<Steps>
  <Step title="Test-Agent-Drawer öffnen">
    Klicken Sie im Agent-Editor auf die Schaltfläche **Agent testen** (oben rechts). Wechseln Sie zum Tab **Telefonanruf**.
  </Step>

  <Step title="Testanruf konfigurieren">
    **Von Nummer**: Wählen Sie eine Ihrer bereitgestellten Nummern als Anrufer-ID

    **Ziel**: Geben Sie Ihre Telefonnummer im E.164-Format ein (`+12125551234`)

    **AMD-Einstellung** (für ausgehende Anrufe):
    - **Textbasiert**: Schnelle Erkennung (~1-2s), 85-90% Genauigkeit
    - **ML-basiert**: Langsamer (~3-4s), 95%+ Genauigkeit
    - Wählen Sie basierend auf Ihren Kampagnenanforderungen
  </Step>

  <Step title="Test starten">
    Klicken Sie auf **Test starten**. Die Plattform wählt sofort Ihr Telefon.

    **Was passiert**:
    - Ihr Telefon klingelt innerhalb von 2-5 Sekunden
    - Nehmen Sie den Anruf entgegen
    - Agent spielt Begrüßung ab
    - Sprechen Sie natürlich, um die Konversation zu testen
  </Step>

  <Step title="Echtzeit-Überwachung">
    Öffnen Sie während des Anrufs **Konversationen → Live-Monitor** in einem anderen Tab, um zu sehen:
    - Live-Transkript
    - Action-Auslöser
    - Anrufqualitätsmetriken
  </Step>

  <Step title="Nach dem Anruf überprüfen">
    Finden Sie nach Abschluss die Konversation unter **Konversationen → Logs**:
    - Vollständige Audioaufzeichnung
    - Transkript mit Zeitstempeln
    - Zielergebnisse
    - Action-Payloads
    - Qualitätsmetriken (Latenz, Jitter, Paketverlust)
  </Step>
</Steps>

## Was zu testen ist

### Audioqualitätsprüfungen

<Accordion title="Klarheit und Aussprache">
  **Achten Sie auf**:
  - Ist die Agentenstimme klar und natürlich?
  - Funktionieren benutzerdefinierte Aussprachen?
  - Gibt es robotische Artefakte oder Störungen?
  - Angemessene Sprechgeschwindigkeit?

  **Testen Sie auf**:
  - Mobiltelefon (Mobilfunknetz)
  - Festnetz (falls verfügbar)
  - VoIP-Telefon (Teams, Zoom Phone)

  Verschiedene Netzwerke haben unterschiedliche Audioqualitätsmerkmale.
</Accordion>

<Accordion title="Echo und Rückkopplung">
  **Testszenarien**:
  1. Freisprechmodus verwenden → Auf Echo achten
  2. Headset/Hörer verwenden → Sollte echofrei sein

  **Probleme**:
  - Echo bei Lautsprecher → Normal, AEC (Echo-Unterdrückung) sollte es handhaben
  - Echo bei Hörer → Problem, Agent-Audioeinstellungen prüfen
  - Rückkopplungsschleife → Eine Audioquelle deaktivieren

  **Konfiguration**: Echo-Unterdrückung aktivieren unter Agent → **Spracheinstellungen → Audioverarbeitung**
</Accordion>

<Accordion title="Latenz und Verzögerungen">
  **Antwortzeit messen**:
  - Hören Sie auf zu sprechen und zählen Sie Sekunden, bis der Agent antwortet
  - Ziel: &lt;1s für beste Erfahrung
  - Akzeptabel: &lt;2s
  - Schlecht: >3s (Kunden werden frustriert)

  **Bei hoher Latenz**:
  - Modellauswahl prüfen (GPT-3.5 schneller als GPT-4)
  - Anweisungslänge reduzieren
  - Wissensdatenbank-Abruf vereinfachen
  - Netzwerkkonnektivität prüfen
</Accordion>

### DTMF (Tastatur) Test

Testen Sie die Telefontastatureingabe, wenn Ihr Agent DTMF verwendet:

<Steps>
  <Step title="DTMF aktivieren">
    In den Agenteneinstellungen → **Erweitert → DTMF-Steuerungen**, Tastaturerkennung aktivieren.
  </Step>

  <Step title="Tastendruck testen">
    Während des Telefonanrufs:
    - Agent sagt: "Drücken Sie 1 für Vertrieb, 2 für Support"
    - Drücken Sie Tasten auf Ihrer Telefontastatur
    - Agent sollte erkennen und reagieren

    Testen Sie alle Tasten: 0-9, *, #
  </Step>

  <Step title="In Logs überprüfen">
    Überprüfen Sie das Konversationslog auf das Feld `dtmf_input`, das zeigt, welche Tasten erkannt wurden.

    Wenn Tasten nicht erkannt werden:
    - Überprüfen Sie, ob der Anbieter RFC 2833 DTMF unterstützt
    - Prüfen Sie, ob DTMF-Einstellungen des Agenten aktiviert sind
    - Versuchen Sie ein anderes Telefon (einige VoIP-Telefone haben Probleme)
  </Step>
</Steps>

### Transfer-Tests

Validieren Sie, dass Anrufweiterleitungen korrekt funktionieren:

<Accordion title="Blinde Weiterleitung">
  **Einrichtung**:
  1. Transfer-Action mit gültiger Zielnummer konfigurieren
  2. Weiterleitung während des Testanrufs auslösen
  3. Agent sollte Weiterleitung ankündigen und Sie verbinden

  **Was zu prüfen ist**:
  - Weiterleitungsankündigung ist klar
  - Anruf wird zur Zielnummer verbunden
  - Audioqualität nach Weiterleitung beibehalten
  - Korrekte Anrufdauerprotokollierung

  **Hinweis**: Verwenden Sie eine zweite Telefonnummer, die Sie kontrollieren, um die Verbindung zu überprüfen.
</Accordion>

<Accordion title="Betreute Weiterleitung">
  **Einrichtung**:
  - Weiterleitungstyp in Action-Konfiguration auf "betreut" setzen
  - Agent sollte zuerst zum Ziel verbinden
  - Ziel vor Verbindung des Anrufers briefen

  **Testablauf**:
  1. Weiterleitung auslösen
  2. Agent setzt Sie in Warteschleife (hört Musik)
  3. Agent verbindet sich mit Ziel
  4. Agent kündigt an: "Ich habe [Kunde] für Sie"
  5. Agent verbindet beide Parteien

  Zuverlässiger als blinde Weiterleitungen, aber langsamer.
</Accordion>

### Mailbox-Erkennung (AMD)

Für ausgehende Kampagnen, testen Sie die Anrufbeantworter-Erkennung:

<Steps>
  <Step title="Ihre Mailbox anrufen">
    Verwenden Sie den Telefontest, um eine Nummer anzurufen, die zur Mailbox geht (Ihre eigene VM oder Test-VM-Nummer).
  </Step>

  <Step title="AMD aktivieren">
    Wählen Sie in den Testeinstellungen entweder textbasierte oder ML-basierte AMD.
  </Step>

  <Step title="Erkennung überprüfen">
    - Anruf sollte Mailbox innerhalb von 1-4 Sekunden erkennen
    - Agent sollte auflegen (oder Nachricht hinterlassen, falls konfiguriert)
    - Überprüfen Sie Logs auf `amd_result: "voicemail"`

    **Genauigkeitstest**:
    - 10 Testanrufe durchführen
    - 5 an echte Person (Sie antworten)
    - 5 an Mailbox
    - Erkennungsgenauigkeit berechnen: Sollte >85% (Text) oder >95% (ML) sein
  </Step>
</Steps>

## Analyse der Telefontestergebnisse

Überprüfen Sie nach jedem Testanruf die Metriken:

### Anrufqualitätsmetriken

Navigieren Sie zu **Konversationen → Logs** → Testanruf auswählen → Tab **Qualität**:

| Metrik | Ziel | Maßnahme bei schlechter Qualität |
|--------|------|----------------------------------|
| **Jitter** | &lt;30ms | Netzwerkstabilität prüfen |
| **Paketverlust** | &lt;1% | Netzwerk oder Anbieter untersuchen |
| **RTT (Round Trip Time)** | &lt;200ms | Geografisches Routing prüfen |
| **MOS (Mean Opinion Score)** | >3.5 | Gesamtqualitätsindikator |

### Transkriptionsgenauigkeit

Vergleichen Sie das Transkript mit dem, was Sie tatsächlich gesagt haben:

- **Wortfehlerrate (WER)**: Sollte &lt;10% sein
- Häufige Fehler: Fachbegriffe, Namen, Akzente
- Beheben Sie mit benutzerdefiniertem Vokabular unter Agent → **Transcriber → Benutzerdefinierte Wörter**

### Action-Ausführung

Überprüfen Sie, ob alle Actions korrekt ausgelöst wurden:
- Zeitstempel prüfen (wurden sie im richtigen Moment ausgelöst?)
- Payloads überprüfen (korrekte Parameter gesendet?)
- Antworten bestätigen (API gab erwartete Daten zurück?)

## Häufige Probleme & Lösungen

<AccordionGroup>
  <Accordion title="Anruf stellt keine Verbindung her">
    **Ursachen**:
    - Nummer noch nicht bereitgestellt (5-10 Min. warten)
    - Falsches Nummernformat (muss E.164 sein: +12125551234)
    - Anbieter blockiert Anrufe (Spam-Score prüfen)
    - SIP-Trunk-Registrierung fehlgeschlagen

    **Lösung**:
    - Überprüfen Sie, ob Nummer "Aktiv"-Status zeigt unter Telefonie → Nummern
    - SIP-Trunk-Registrierungslogs prüfen
    - Mit anderer Zielnummer testen
  </Accordion>

  <Accordion title="Einseitiges Audio">
    **Symptom**: Sie hören den Agenten, aber Agent hört Sie nicht (oder umgekehrt)

    **Ursachen**:
    - Firewall blockiert RTP-Pakete
    - NAT-Traversal-Problem
    - Codec-Nichtübereinstimmung

    **Lösung**:
    - TURN-Relay in Agenteneinstellungen aktivieren
    - Prüfen Sie, ob Firewall UDP-Ports 10000-60000 zulässt
    - Anderen Codec ausprobieren (Opus → PCMU)
  </Accordion>

  <Accordion title="Schlechte Audioqualität">
    **Ursachen**:
    - Netzwerküberlastung
    - Geringe Bandbreite
    - Paketverlust
    - Falsche Codec-Einstellungen

    **Lösung**:
    - Anrufqualitätsmetriken in Logs prüfen
    - Zu höherwertigen Codec wechseln
    - Zu anderer Zeit testen (weniger Netzwerklast)
    - Anbieter kontaktieren, wenn Problem anhält
  </Accordion>
</AccordionGroup>

## Telefontest-Checkliste

Vervollständigen Sie vor dem Launch in die Produktion diese Telefontest-Checkliste:

- [ ] Von Mobiltelefon angerufen - Audio in beiden Richtungen klar
- [ ] Von Festnetz angerufen (falls zutreffend) - Audio klar
- [ ] In lauter Umgebung getestet - Agent transkribiert genau
- [ ] DTMF-Eingabe getestet - alle Tasten korrekt erkannt
- [ ] Weiterleitung ausgelöst - erfolgreich mit Ziel verbunden
- [ ] AMD getestet - Mailbox genau erkannt
- [ ] Agent mitten im Satz unterbrochen - Barge-in funktioniert
- [ ] Still geblieben - Inaktivitätsbehandlung ausgelöst
- [ ] Alle benutzerdefinierten Actions ausgelöst - erfolgreich ausgeführt
- [ ] KB-Fragen gestellt - korrekte Informationen abgerufen
- [ ] Anrufqualitätsmetriken überprüft - alle innerhalb der Ziele
- [ ] Transkriptgenauigkeit geprüft - &lt;10% Wortfehlerrate
