## Debugging-Philosophie

Wenn sich Ihr Agent unerwartet verhält, spart systematisches Debugging Zeit. Folgen Sie diesem Prozess:

1. **Reproduzieren** - Können Sie das Problem konsistent reproduzieren?
2. **Isolieren** - Liegt es an Instruktionen, Integrationen, Modell oder Infrastruktur?
3. **Inspizieren** - Überprüfen Sie Logs, Traces und Metriken auf Hinweise
4. **Beheben** - Wenden Sie die kleinste Änderung an, die das Problem löst
5. **Validieren** - Testen Sie, um zu bestätigen, dass die Lösung funktioniert

## Häufige Debugging-Szenarien

### Agent gibt falsche Antworten

<Steps>
  <Step title="Instruktionen überprüfen">
    Überprüfen Sie Agent → **Conversation → Instructions**:
    - Sind die Instruktionen klar und spezifisch?
    - Gibt es widersprüchliche Regeln?
    - Hat der Agent Kontext für die Frage?
  </Step>

  <Step title="Knowledge Base verifizieren">
    - Ist die korrekte Antwort in Ihrer KB?
    - Testen Sie die KB-Suche manuell:
      ```bash
      curl -X POST https://api.itellico.ai/v1/knowledge-bases/kb_123/search \
           -H "Authorization: Bearer YOUR_API_KEY" \
           -d '{"query": "your test query"}'
      ```
    - Überprüfen Sie das Abrufen in Logs → Abschnitt **Knowledge Retrieved**
  </Step>

  <Step title="Gesprächsprotokoll überprüfen">
    - Welchen Kontext hatte der Agent bei der Beantwortung?
    - Wurde relevanter KB-Inhalt abgerufen?
    - Hat der Agent den Instruktionsvorlagen gefolgt?
  </Step>

  <Step title="Isoliert testen">
    Erstellen Sie einen minimalen Test:
    - Einfache Instruktion
    - Klare Frage
    - Erwarteter KB-Inhalt

    Wenn es isoliert funktioniert, liegt das Problem bei Kontext/Komplexität.
  </Step>
</Steps>

### Actions werden nicht ausgelöst

<Accordion title="API-Action wird nicht ausgelöst">
  **In Gesprächsprotokollen überprüfen**:
  1. Gehen Sie zu Gespräch → **Actions**-Tab
  2. Ist die Action aufgelistet? Falls nicht, wurde sie nie ausgelöst
  3. Falls aufgelistet, überprüfen Sie den Status: Success vs Failed

  **Warum Actions nicht ausgelöst werden**:
  - Agent-Instruktionen erwähnen nicht, wann sie verwendet werden soll
  - Benutzer hat keine erforderlichen Informationen bereitgestellt
  - Action-Auslösebedingung nicht erfüllt

  **Behebung**:
  - Fügen Sie klare Instruktion hinzu: "Wenn der Kunde nach dem Bestellstatus fragt, verwenden Sie die getOrderStatus-Action"
  - Stellen Sie sicher, dass der Agent zuerst erforderliche Parameter sammelt
  - Testen Sie die Auslösephrase im Simulator
</Accordion>

<Accordion title="API gibt Fehler zurück">
  **Action-Details in Logs überprüfen**:
  - Gesendeter Request (Payload-Format korrekt?)
  - Empfangene Response (Fehlercode? Nachricht?)
  - Auth-Header (API-Key gültig?)

  **Häufige Fehler**:
  - `401 Unauthorized` → API-Key überprüfen
  - `404 Not Found` → Endpoint-URL verifizieren
  - `408 Timeout` → API zu langsam, Timeout erhöhen
  - `500 Internal Server Error` → Problem mit externer API

  **API manuell testen**:
  ```bash
  # Request aus Logs kopieren und mit curl testen
  curl -X POST https://api.example.com/endpoint \
       -H "Authorization: Bearer YOUR_KEY" \
       -H "Content-Type: application/json" \
       -d '{"param": "value"}'
  ```

  Wenn manueller Aufruf funktioniert, liegt das Problem in der Action-Konfiguration.
</Accordion>

### Transkriptionsfehler

<Accordion title="Agent verhört Wörter">
  **Transkriptgenauigkeit überprüfen**:
  - Anrufaufzeichnung anhören
  - Mit Transkripttext vergleichen
  - Verhörte Wörter identifizieren (Fachbegriffe? Namen? Akzente?)

  **Lösungen**:

  **1. Benutzerdefiniertes Vokabular hinzufügen**:
  ```json
  // Agent-Einstellungen → Transcriber → Custom Vocabulary
  {
    "custom_words": [
      {"word": "itellicoAI", "pronunciation": "eye-TELL-ih-koh AI"},
      {"word": "API", "pronunciation": "A P I"}
    ]
  }
  ```

  **2. Anderen Transcriber ausprobieren**:
  - Deepgram: Am besten für Akzente, Lärm
  - Azure: Am besten für Fachbegriffe
  - Whisper: Am besten für mehrsprachig

  **3. Audioqualität verbessern**:
  - Rauschunterdrückung aktivieren
  - Höhere Bitrate-Codec verwenden (Opus 64kbps)
  - In ruhigerer Umgebung testen
</Accordion>

### Agent wiederholt sich oder läuft in Schleife

<Accordion title="Agent stellt dieselbe Frage mehrmals">
  **Diagnose**:
  1. Transkript überprüfen - wo beginnt die Schleife?
  2. Welche Benutzerantwort löst die Wiederholung aus?
  3. Schlägt die Transkription fehl? (Agent hat Antwort nicht "gehört")

  **Lösungen**:

  **1. Instruktionen zur Schleifenvermeidung hinzufügen**:
  ```markdown
  Wenn Sie dieselbe Frage zweimal gestellt haben, ohne eine klare Antwort zu erhalten:
  1. Formulieren Sie die Frage um
  2. Wenn immer noch unklar, bieten Sie an, zu einem Menschen weiterzuleiten

  Stellen Sie dieselbe Frage nie mehr als 3 Mal.
  ```

  **2. Strukturierte Eingabe verwenden (DTMF)**:
  Für kritische Informationen, Tastatur verwenden:
  ```markdown
  "Welche Abteilung? Drücken Sie 1 für Vertrieb, 2 für Support."
  ```

  Zuverlässiger als Spracherkennung.

  **3. Transkription überprüfen**:
  Wenn der Agent die Antwort nicht "gehört" hat, verbessern Sie die Audioqualität oder Transcriber-Einstellungen.
</Accordion>

## Debugging mit Logs

### Gesprächsprotokolle lesen

Jedes Gesprächsprotokoll enthält:

<Tabs>
  <Tab title="Transkript">
    **Worauf zu achten ist**:
    - Transkriptionsgenauigkeit (mit Audio vergleichen)
    - Relevanz der Agent-Antwort
    - Gesprächsfluss (natürlich vs. holprig)
    - Indikatoren für Zielerreichung

    **Zeitstempel**: Prüfen Sie, ob Verzögerungen zwischen Turns auf Latenzprobleme hinweisen
  </Tab>

  <Tab title="Actions">
    **Details**:
    - Welche Actions ausgelöst wurden
    - Eingabeparameter
    - API-Request/Response
    - Ausführungszeit
    - Success/Failure-Status

    **Fehler debuggen**: Fehlermeldungen weisen normalerweise auf die Grundursache hin
  </Tab>

  <Tab title="Knowledge">
    **Was abgerufen wurde**:
    - Verwendete Suchanfrage
    - Abgerufene KB-Chunks
    - Ähnlichkeits-Scores
    - In Antwort verwendeter Inhalt

    **"Ich weiß nicht" debuggen**: Prüfen Sie, ob relevanter Inhalt abgerufen, aber nicht verwendet wurde, oder ob das Abrufen vollständig fehlgeschlagen ist
  </Tab>

  <Tab title="Goals">
    **Ergebnisse**:
    - Welche Ziele erreicht wurden
    - Konfidenz-Scores
    - Extrahierte Entitäten (Daten, Namen, etc.)

    **Fehlklassifizierung debuggen**: Gespräch überprüfen - hätte das Ziel erreicht werden sollen?
  </Tab>

  <Tab title="Quality">
    **Metriken**:
    - Audioqualität (Jitter, Paketverlust, MOS-Score)
    - Response-Latenz
    - Anrufdauer
    - Unterbrechungshäufigkeit

    **Performance-Probleme**: Latenz- und Qualitätsmetriken überprüfen
  </Tab>
</Tabs>

### Fehlerprotokoll-Analyse

Fehler filtern nach:
- Zeitbereich (letzte 24h, letzte Woche)
- Fehlertyp (API, Transkription, System)
- Agent-ID (spezifischer Agent)
- Schweregrad (kritisch, Warnung, Info)

**Häufige Muster**:
- Derselbe Fehler wiederholt sich → Systemisches Problem (Konfiguration, Integration)
- Fehler-Spitzen zu bestimmter Zeit → Kapazitäts- oder externes Service-Problem
- Fehler bei spezifischem Agent → Agent-spezifisches Konfigurationsproblem

## Erweitertes Debugging

### Debugging des Modellverhaltens

<Accordion title="Modellantworten vergleichen">
  Testen Sie dasselbe Szenario mit verschiedenen Modellen:

  **Schritte**:
  1. Agent klonen
  2. Modell ändern (GPT-4 → Claude, etc.)
  3. Identisches Testszenario ausführen
  4. Antworten vergleichen

  **Wann nützlich**:
  - Agent gibt unerwartete Antworten
  - Testen, ob Modell-Upgrade Qualität verbessert
  - Debugging von Halluzinationen oder falschen Fakten
</Accordion>

<Accordion title="Prompt-Engineering-Iteration">
  **Systematisches Testen**:
  1. Erstellen Sie 5-10 Testszenarien (in Test Scenarios gespeichert)
  2. Führen Sie alle Szenarien mit aktuellen Instruktionen aus
  3. Fehler dokumentieren
  4. Instruktionen ändern, um Fehler zu beheben
  5. Alle Szenarien erneut ausführen
  6. Wiederholen, bis alle bestehen

  **A/B-Test-Instruktionen**:
  - Erstellen Sie zwei Agent-Versionen mit unterschiedlichen Instruktionen
  - Leiten Sie 50% des Traffics zu jedem
  - Vergleichen Sie Zielerreichungsraten
  - Bester Agent gewinnt
</Accordion>

### Integrations-Debugging

<Accordion title="APIs isoliert testen">
  **Agent aus der Gleichung entfernen**:
  1. API-Request aus Logs kopieren
  2. Mit curl/Postman testen
  3. Response-Format gegen Erwartungen verifizieren
  4. Auth bestätigen funktioniert

  Wenn API isoliert funktioniert, aber vom Agent fehlschlägt:
  - IP-Allowlisting überprüfen
  - Timeout-Einstellungen verifizieren (Agent könnte Timeout vor API-Antwort)
  - Request-Payload-Formatierung überprüfen
</Accordion>

<Accordion title="API-Antworten mocken">
  Zum Testen ohne echte APIs zu verwenden:

  **Mock-Server verwenden**:
  ```javascript
  // mockapi.example.com/order-status
  {
    "order_id": "12345",
    "status": "shipped",
    "eta": "2024-01-15"
  }
  ```

  Agent-Action auf Mock-URL zeigen, Gesprächsfluss ohne externe Abhängigkeiten testen.
</Accordion>

## Debugging-Checkliste

Beim Debugging eines Problems:

- [ ] Können Sie das Problem konsistent reproduzieren?
- [ ] Gesprächsprotokoll für den fehlgeschlagenen Anruf überprüft?
- [ ] Fehlerprotokolle auf verwandte Fehler überprüft?
- [ ] Agent-Instruktionen auf Klarheit überprüft?
- [ ] Im Web-Simulator getestet (schnellere Iteration)?
- [ ] Problem isoliert (Instruktionen vs. Integration vs. Modell)?
- [ ] Minimale Lösung angewendet (eine Sache geändert)?
- [ ] Erneut getestet, um Lösung zu bestätigen?
- [ ] Auf Regressionen getestet (nichts anderes kaputt gemacht)?
- [ ] Problem und Lösung dokumentiert?
