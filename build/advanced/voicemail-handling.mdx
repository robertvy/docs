## Overview

Answering Machine Detection (AMD) enables your AI agents to automatically detect when outbound calls reach voicemail instead of a live person, allowing you to configure different behaviors for each scenario. This critical feature prevents wasted agent time talking to answering machines, reduces negative customer experiences from inappropriate voicemail interactions, and optimizes campaign efficiency by focusing resources on live conversations.

Effective voicemail handling transforms your outbound campaigns from basic autodialers into intelligent communication systems that respect customer preferences, maintain professional brand standards, and maximize connection rates with decision-makers.

<Note>
AMD configuration is primarily accessed during campaign creation and in campaign settings. Different AMD methods (text-based vs. ML-based) offer trade-offs between speed, accuracy, and use case fit.
</Note>

## What is Answering Machine Detection?

### The Challenge

When making outbound calls, you encounter two possible scenarios:

**Scenario 1: Live Answer**
```
Phone rings → Person answers → "Hello?"
→ Agent should engage in conversation
→ Full agent capabilities needed
```

**Scenario 2: Voicemail**
```
Phone rings → Voicemail system answers → "You've reached John Smith..."
→ Agent should either leave brief message or hang up
→ Don't waste time with full conversation script
→ Don't create awkward interaction talking over voicemail greeting
```

**The problem:** How does the agent know which scenario occurred?

### AMD Solution

AMD analyzes the audio in the first few seconds after call connection to determine if a human or machine answered:

**Detection process:**
```
1. Call connects
2. AMD analyzes audio (0.5-3 seconds depending on method)
3. Classification: HUMAN or MACHINE
4. Agent executes appropriate behavior
```

**Benefits:**
- **Efficiency:** Don't waste agent time on voicemail
- **Professionalism:** Leave appropriate messages, not awkward partial scripts
- **Higher connect rates:** Focus retry attempts on different times rather than immediate reattempts
- **Better analytics:** Separate "reached voicemail" from "no answer" in reporting

## AMD Methods

### Overview of Detection Types

<CardGroup cols={2}>
  <Card title="Text-Based AMD" icon="text">
    Fast, keyword-based detection using speech-to-text
  </Card>
  <Card title="ML-Based AMD" icon="brain">
    Intelligent pattern recognition using machine learning models
  </Card>

### Text-Based AMD

**How it works:**
```
1. Call connects
2. Speech-to-text transcribes first few seconds
3. Scans for voicemail keywords:
   - "You've reached"
   - "Leave a message"
   - "Not available"
   - "Voicemail"
   - "After the beep"
4. If keywords detected → MACHINE
5. If no keywords in first 2-3 seconds → HUMAN
```

**Characteristics:**

<AccordionGroup>
  <Accordion title="Speed" icon="gauge-high">
    **Very fast:** ~500ms-1.5 seconds

    Begins processing immediately upon connection, provides classification quickly

    **Best for:** Time-sensitive campaigns where speed matters
  </Accordion>

  <Accordion title="Accuracy" icon="chart-line">
    **Good accuracy:** 85-92% in typical scenarios

    **High accuracy when:**
    - Standard voicemail greetings with common phrases
    - Clear audio quality
    - English language voicemails
    - B2B environments with professional greetings

    **Lower accuracy when:**
    - Custom greetings without standard keywords
    - Short greetings ("Hi, leave a message" - very brief)
    - Non-English voicemails
    - Background noise interfering with transcription
  </Accordion>

  <Accordion title="Best Use Cases" icon="briefcase">
    **Ideal for:**
    - **B2B campaigns** - Business voicemails typically use standard phrasing
    - **High-volume campaigns** - Speed advantage compounds across thousands of calls
    - **Budget-conscious deployments** - Lower computational cost
    - **English-language markets** - Keyword detection optimized for English

    **Example scenarios:**
    - Sales outreach to business phone numbers
    - Appointment reminders to office lines
    - B2B lead qualification campaigns
  </Accordion>

  <Accordion title="Limitations" icon="triangle-exclamation">
    **May struggle with:**
    - Personal, creative voicemail greetings ("Hey, it's Mike, you know what to do")
    - Very short greetings
    - Non-English greetings
    - Greetings that sound conversational ("Hello? Hello? Just kidding, leave a message")
    - Background music or noise in greeting

    **False positives:** Human who starts with "You've reached..." might be misclassified

    **False negatives:** Voicemail without keywords might be classified as human
  </Accordion>
</AccordionGroup>

### ML-Based AMD

**How it works:**
```
1. Call connects
2. ML model analyzes audio characteristics:
   - Speech patterns and cadence
   - Intonation and naturalness
   - Background characteristics
   - Timing and pacing
   - Acoustic features
3. Classification based on learned patterns
4. Returns: HUMAN or MACHINE with confidence score
```

**Characteristics:**

<AccordionGroup>
  <Accordion title="Speed" icon="gauge">
    **Moderate speed:** ~1.5-3 seconds

    Requires slightly more audio to analyze patterns

    **Trade-off:** Slower than text-based, but higher accuracy
  </Accordion>

  <Accordion title="Accuracy" icon="chart-line">
    **Higher accuracy:** 92-97% in most scenarios

    **Advantages over text-based:**
    - Doesn't rely on specific keywords
    - Detects machine-like speech patterns regardless of content
    - Better with non-standard greetings
    - Works across languages
    - Handles creative/personal greetings better

    **Still challenging:**
    - Very short greetings (< 1 second of audio)
    - Highly degraded audio quality
    - Extreme edge cases (person mimicking voicemail)
  </Accordion>

  <Accordion title="Best Use Cases" icon="briefcase">
    **Ideal for:**
    - **Consumer campaigns** - Personal voicemails with creative greetings
    - **Multi-language campaigns** - Not dependent on English keywords
    - **Quality-focused campaigns** - When accuracy is more important than speed
    - **Complex markets** - Mixed business/personal numbers

    **Example scenarios:**
    - Consumer sales calls
    - Political campaigns
    - Non-profit fundraising
    - Healthcare outreach
    - Multi-language support campaigns
  </Accordion>

  <Accordion title="Advantages" icon="check">
    **Handles well:**
    - Creative personal greetings
    - Short greetings
    - Non-English voicemails
    - Greetings without standard keywords
    - Background music or sound effects
    - Natural conversational-sounding greetings

    **Robust across:**
    - Different languages
    - Regional accents
    - Various voicemail systems
    - Custom greetings
  </Accordion>
</AccordionGroup>

### Choosing the Right AMD Method

**Decision matrix:**

| Factor | Text-Based AMD | ML-Based AMD |
|--------|---------------|--------------|
| **Campaign Type** | B2B | Consumer |
| **Priority** | Speed | Accuracy |
| **Typical Greeting Style** | Standard/Professional | Creative/Personal |
| **Language** | English | Multi-language |
| **Volume** | Very high | Moderate-high |
| **Voicemail Keyword Usage** | Common | Varies widely |
| **Cost Sensitivity** | Budget-conscious | Quality-focused |

**Recommendation:**
```
Use Text-Based AMD if:
✓ Primarily B2B calling
✓ English language only
✓ Need fastest possible detection
✓ Standard professional greetings expected
✓ Budget/cost is priority

Use ML-Based AMD if:
✓ Consumer calling
✓ Multi-language support needed
✓ Creative/personal greetings common
✓ Accuracy is critical
✓ Willing to accept slightly slower detection
```

<Tip>
When in doubt, start with ML-based AMD for higher accuracy. The 1-2 second speed difference is minimal compared to the cost of misclassification (talking to voicemail or hanging up on humans).
</Tip>

## Configuring AMD

### Setting AMD Method (Campaign Creation)

AMD is configured when creating a new outbound campaign:

<Steps>
  <Step title="Navigate to Campaign Creation">
    Go to **Campaigns** section

    Click **Create Campaign** (or **Create First Campaign**)
  </Step>

  <Step title="Select AMD Method">
    In the campaign creation form, find **Answering Machine Detection (AMD)** field

    **Options:**
    - **Text-based** - Fast keyword detection
    - **ML-based** - Machine learning pattern recognition

    Select the method appropriate for your use case
  </Step>

  <Step title="Complete Campaign Setup">
    Fill in other required fields:
    - Campaign Name
    - Agent
    - Phone Number

    Click **Create Campaign**
  </Step>

  <Step title="Configure Voicemail Behavior">
    After campaign creation, configure how agent handles detected voicemail:
    - Leave message
    - Hang up silently
    - Schedule retry

    (Detailed in sections below)
  </Step>
</Steps>

### Changing AMD Method (Existing Campaign)

To change AMD method for existing campaign:

1. Navigate to campaign **Settings** tab
2. Locate **Answering Machine Detection** section
3. Select different AMD method
4. Save changes
5. Note: Change applies to future calls only, not in-progress calls

<Warning>
Changing AMD method mid-campaign may affect analytics consistency. Consider creating a new campaign if you need to A/B test AMD methods.
</Warning>

## Voicemail Message Configuration

### Configuring Agent Behavior for Voicemail

When AMD detects voicemail, configure how your agent responds:

**Option 1: Leave voicemail message**
```
Instructions example:

When voicemail detected:
1. Wait for beep (listen for tone indicating recording started)
2. Deliver brief, professional message:

   "Hi, this is [Agent Name] calling from [Company Name].
   I'm reaching out regarding [brief reason].
   Please call us back at [phone number].
   Again, that's [phone number slowly].
   Thank you, and have a great day."

3. End call after message delivery
```

**Option 2: Hang up silently**
```
Instructions example:

When voicemail detected:
1. End call immediately without leaving message
2. Mark contact for retry at different time
3. Log "reached voicemail" in campaign analytics

Use when:
- Multiple retry attempts planned (will try different times)
- Voicemail messages not effective for campaign type
- Avoiding voicemail spam concerns
```

**Option 3: Conditional based on attempt number**
```
Instructions example:

{% if campaign_contact_attempt_count == 1 %}
First attempt - hang up silently, will retry
{% elif campaign_contact_attempt_count == 2 %}
Second attempt - hang up silently, will retry once more
{% elif campaign_contact_attempt_count == 3 %}
Final attempt - leave voicemail message:
"This is our final attempt to reach you regarding [topic].
Please call [number] if you're interested."
{% endif %}
```

### Beep Detection

**Challenge:** Voicemail systems play greeting before the recording beep. Agent should wait for beep before speaking to ensure message is recorded.

**Beep detection process:**
```
1. AMD detects MACHINE
2. Agent enters "wait for beep" mode
3. Listens for characteristic tone (usually 1-2 seconds of 1000Hz tone)
4. Beep detected → Begin speaking message
5. Beep not detected after 10 seconds → Assume recording started, begin message
```

**Instructions for beep handling:**
```
When voicemail is detected:

1. Listen carefully for the recording beep (usually a single tone)
2. Wait up to 8 seconds for the beep
3. Once beep is heard, immediately begin your message
4. If no beep detected after 8 seconds, begin message anyway
   (some systems don't have distinct beep)

Do NOT start speaking during the voicemail greeting - wait for
the beep so your message is actually recorded.
```

### Crafting Effective Voicemail Messages

**Best practices for agent voicemail scripts:**

<AccordionGroup>
  <Accordion title="Keep It Brief" icon="clock">
    **Target length:** 15-25 seconds maximum

    **Why:** Long voicemails are often deleted without full listening

    **Structure:**
    ```
    1. Greeting (2 seconds): "Hi, this is [Name] from [Company]"
    2. Reason (5 seconds): "I'm calling about [specific topic]"
    3. Call to action (5 seconds): "Please call me back at [number]"
    4. Repeat number slowly (5 seconds): "Again, that's [number]"
    5. Closing (2 seconds): "Thank you, goodbye"

    Total: ~20 seconds
    ```
  </Accordion>

  <Accordion title="Speak Clearly and Slowly" icon="volume">
    **Especially important for:**
    - Company name
    - Your name
    - Phone number (say each digit clearly)
    - Reason for call

    **Instructions example:**
    ```
    When leaving voicemail:
    - Speak at 90% of normal conversation speed
    - Pause briefly between name and company
    - When stating phone number, pause between digit groups:
      "Five five five (pause) twelve thirty-four"
    - Enunciate clearly, especially for names/numbers
    ```
  </Accordion>

  <Accordion title="Include Specific Information" icon="circle-info">
    **Make it relevant and actionable:**

    **Good:**
    ```
    "Hi Sarah, this is Mike from Acme Software. I'm calling about
    the product demo you requested on our website last Tuesday.
    I have your demo link ready. Please call me back at
    555-123-4567. That's 555-123-4567. Thanks!"
    ```

    **Bad:**
    ```
    "This is a call from Acme Software. Please call us back."
    [No name, no specific reason, no number]
    ```

    **Include:**
    - Agent name (builds trust)
    - Company name
    - Specific reason (not generic)
    - Clear call to action
    - Callback number (slowly, twice)
  </Accordion>

  <Accordion title="Set Appropriate Tone" icon="face-smile">
    **Match tone to campaign type:**

    **Professional (B2B):**
    ```
    "Good morning, this is Jennifer from TechCorp Solutions.
    I'm reaching out regarding your inquiry about our enterprise
    platform. I'd like to schedule a brief demo at your convenience.
    Please call me at 555-123-4567. Again, 555-123-4567.
    Looking forward to speaking with you."
    ```

    **Friendly (Consumer):**
    ```
    "Hi! This is Mark calling from Sunshine Solar. We're offering
    free home energy assessments in your neighborhood this week.
    I'd love to see if we can help lower your electric bill.
    Give me a call back at 555-123-4567. That's 555-123-4567.
    Talk to you soon!"
    ```

    **Urgent (Time-sensitive):**
    ```
    "Hi, this is calling from City Hospital regarding your
    appointment tomorrow at 2 PM. We need to confirm. Please
    call us back as soon as possible at 555-123-4567.
    Again, 555-123-4567. Thank you."
    ```
  </Accordion>

  <Accordion title="Avoid Common Mistakes" icon="triangle-exclamation">
    **Don't:**
    - ❌ Speak too fast (especially phone number)
    - ❌ Leave voicemail longer than 30 seconds
    - ❌ Forget to include callback number
    - ❌ Say number only once (always repeat)
    - ❌ Use overly salesy language
    - ❌ Include too much information (keep it simple)
    - ❌ Sound robotic or unnatural
    - ❌ Start speaking before beep (message won't be recorded)

    **Do:**
    - ✅ Speak clearly at moderate pace
    - ✅ Include name, company, reason, number
    - ✅ Repeat number twice, slowly
    - ✅ Keep message under 25 seconds
    - ✅ Sound professional and friendly
    - ✅ Wait for beep before speaking
    - ✅ End with clear call to action
  </Accordion>
</AccordionGroup>

## AMD in Campaign Workflows

### Campaign-Level AMD Strategy

**Integrating AMD with campaign retry logic:**

```
Campaign AMD Configuration:

Attempt 1:
- AMD detects voicemail → Hang up
- Schedule retry for +24 hours, different time of day
- Reasoning: Try to reach person at different time

Attempt 2:
- AMD detects voicemail → Hang up
- Schedule retry for +48 hours, different time/day
- Reasoning: One more attempt at different time

Attempt 3 (Final):
- AMD detects voicemail → Leave message
- Include "final attempt" language
- Mark contact as "Voicemail - Message Left"
- No additional retries

This strategy:
- Maximizes chance of live conversation (attempts 1-2)
- Leaves professional impression (attempt 3 message)
- Respects customer's time (limits voicemail spam)
- Provides closure (clear final message)
```

### AMD and Contact Status

**How AMD affects contact lifecycle:**

```
Contact Status Flow with AMD:

Pending
  ↓
Dialing
  ↓
AMD Classification
  ├─ HUMAN DETECTED
  │    → Called (conversation proceeds)
  │    → Agent achieves goal or not
  │    → Completed or Retry
  │
  └─ MACHINE DETECTED
       → Mark as "Voicemail"
       ├─ Leave message → Retry or Completed
       └─ Hang up → Retry (different time)
```

**Analytics impact:**
- Separate "reached voicemail" from "no answer" in reporting
- Track voicemail rate by time of day (optimize calling schedule)
- Measure callback rate from voicemail messages
- Identify contacts who always go to voicemail (adjust strategy)

### Optimizing Based on AMD Data

**Use AMD analytics to improve campaigns:**

<AccordionGroup>
  <Accordion title="Identify Best Calling Times" icon="clock">
    **Analysis:**
    Review "Human Answer Rate by Time" in campaign reporting

    **Metrics to track:**
    - Voicemail rate by hour of day
    - Voicemail rate by day of week
    - Human answer rate by time slot

    **Optimization:**
    ```
    If data shows:
    - 8-10 AM: 70% voicemail rate
    - 12-2 PM: 60% voicemail rate
    - 4-6 PM: 40% voicemail rate ← Best time

    Action:
    - Adjust campaign business hours to focus on 4-6 PM
    - Reduce calling during 8-10 AM
    ```

  </Accordion>

  <Accordion title="Segment by Voicemail Behavior" icon="users">
    **Create contact segments:**

    **"Always Voicemail" segment:**
    - 3+ attempts, all reached voicemail
    - Strategy: Try drastically different time (early morning/evening)
    - Or: Remove from campaign, try different channel (email)

    **"Sometimes Answers" segment:**
    - Mix of voicemail and human answers
    - Strategy: Continue normal retry cadence
    - Higher priority than "always voicemail"

    **"Usually Answers" segment:**
    - Majority of attempts reach human
    - Strategy: Standard retry schedule works well
  </Accordion>

  <Accordion title="Test Message Effectiveness" icon="flask">
    **A/B test voicemail messages:**

    **Test variables:**
    - Message length (15 sec vs. 25 sec)
    - Tone (professional vs. casual)
    - Content (specific vs. generic reason)
    - Number repetition (once vs. twice)

    **Measure:**
    - Callback rate
    - Conversion rate from voicemail
    - Time to callback

    **Example:**
    ```
    Version A (Control):
    "Hi, this is Mike from Acme. Please call back at 555-1234."
    Callback rate: 3%

    Version B (Test):
    "Hi, this is Mike from Acme regarding your software inquiry
    from our website. I have information ready for you.
    Call me at 555-1234. That's 555-1234."
    Callback rate: 7%

    Result: Version B performs 2.3x better, make it standard
    ```

### AMD Test Plan

<Steps>
  <Step title="Test Text-Based AMD">

       - Your own voicemail (standard greeting)

    1. Add test numbers to campaign
    2. Launch campaign
    3. Let calls reach voicemail
    4. Review campaign logs for AMD classification

    **Validation:**
    - Text-based AMD correctly identifies voicemail

  <Step title="Test ML-Based AMD">

    1. Add numbers to campaign
    2. Launch campaign

    - ML-based AMD handles varied greetings correctly

  <Step title="Test Human vs. Machine Classification">

    1. Have team member answer test call

## Best Practices

<AccordionGroup>
  <Accordion title="Choose AMD Method Based on Campaign Type" icon="filter">
    **B2B campaigns:**
    - Use text-based AMD for speed
    - Professional voicemails typically use standard keywords
    - High volume benefits from faster detection

    **Consumer campaigns:**
    - Use ML-based AMD for accuracy
    - Personal greetings vary widely
    - Accuracy more important than speed difference

    **Multi-language campaigns:**
    - Always use ML-based AMD
    - Text-based keyword detection doesn't work across languages
  </Accordion>

  <Accordion title="Don't Spam Voicemail" icon="ban">
    **Avoid leaving message on every attempt**

    **Bad strategy:**
    ```
    Attempt 1: Voicemail → Leave message
    Attempt 2 (24 hours later): Voicemail → Leave message
    Attempt 3 (48 hours later): Voicemail → Leave message

    Result: Customer has 3 voicemails from same company in 3 days
    → Annoying, unprofessional, spam-like
    ```

    **Good strategy:**
    ```
    Attempt 1: Voicemail → Hang up (try different time)
    Attempt 2: Voicemail → Hang up (one more try)
    Attempt 3: Voicemail → Leave single professional message

    Result: Customer has 1 voicemail, not overwhelmed
    → Professional, respectful of their time
    ```
  </Accordion>

  <Accordion title="Optimize Calling Times Based on Data" icon="chart-line">
    **Use AMD data to find best times:**

    **Track voicemail rate by time:**
    - High voicemail rate (70%+) → Bad time to call
    - Moderate voicemail rate (40-60%) → Okay time
    - Low voicemail rate (20-30%) → Best time to call

    **Adjust campaign schedule:**
    - Focus calling hours on low-voicemail times
    - Reduce or eliminate high-voicemail time slots
    - Schedule retries for optimal times

    **Example optimization:**
    ```
    Original schedule: 9 AM - 5 PM (uniform distribution)
    Voicemail analysis shows:
    - 9-11 AM: 75% voicemail (people in meetings)
    - 11 AM-1 PM: 60% voicemail (lunch)
    - 1-3 PM: 35% voicemail (best window)
    - 3-5 PM: 50% voicemail

    Optimized schedule: Focus 70% of calls in 1-3 PM window
    Result: 40% reduction in voicemail rate, more live conversations
    ```
  </Accordion>

  <Accordion title="Test Both AMD Methods" icon="flask">
    **Before committing to large campaign:**

    1. Create two small test campaigns (100 contacts each)
    2. One with text-based AMD, one with ML-based AMD
    3. Use same agent, same contacts type, same time
    4. Compare results:
       - Classification accuracy
       - Human answer rate
       - False positive/negative rate
       - Speed difference impact
    5. Choose method with best results for your specific use case

    **Don't assume:** Test with your actual contact list and geography
  </Accordion>

  <Accordion title="Monitor and Improve Messages" icon="message">
    **Treat voicemail messages as ongoing optimization:**

    **Quarterly review:**
    1. Listen to sample of voicemail messages left
    2. Review callback rate from voicemail
    3. Test message variations
    4. Update agent instructions based on learnings

    **A/B test systematically:**
    - Test one variable at a time (length, tone, content)
    - Measure callback rate and conversion
    - Implement winning variations
    - Continue iterating
  </Accordion>

  <Accordion title="Respect Do Not Call and Voicemail Preferences" icon="shield">
    **Compliance and ethics:**

    **Do Not Call lists:**
    - Scrub voicemail-only contacts against DNC lists
    - Even if leaving message (not live call), respect DNC

    **Voicemail preferences:**
    - If contact calls back and asks not to leave voicemails, update record
    - Track "no voicemail" preference in contact metadata
    - Instructions: Check preference before leaving message

    **Frequency caps:**
    - Don't leave more than 1-2 voicemails per campaign
    - Space voicemails at least 7 days apart
    - After 2-3 voicemails with no response, stop

    **Example instruction:**
    ```
    {% if contact.no_voicemail_preference == true %}
    Do not leave voicemail message. Hang up and schedule retry.
    {% elif campaign_voicemails_left >= 2 %}
    Maximum voicemails reached for this contact. Do not leave message.
    {% else %}
    Proceed with voicemail message.
    {% endif %}
    ```
  </Accordion>
</AccordionGroup>

## Troubleshooting

<AccordionGroup>
  <Accordion title="False Positives (Human Classified as Machine)" icon="user-xmark">
    **Symptoms:** Live person answers but AMD classifies as voicemail

    **Possible causes:**
    1. Person answers with greeting that sounds like voicemail
    2. Text-based AMD detects keywords ("You've reached John")
    3. Very scripted/formal greeting sounds machine-like

    **Impact:**
    - Agent hangs up on live person or leaves voicemail message
    - Lost opportunity for conversation
    - Poor customer experience

    **Solutions:**
    - Switch to ML-based AMD (better at distinguishing patterns)
    - Add fallback: If customer responds during "voicemail message," switch to conversation
    - Instruction example:
      ```
      If delivering voicemail message and customer interrupts
      or responds, immediately switch to conversation mode:
      "Oh, hi! I apologize, I thought I reached voicemail.
      Let me start over - this is [name] from [company]..."
      ```
  </Accordion>

  <Accordion title="False Negatives (Machine Classified as Human)" icon="robot">
    **Symptoms:** Voicemail answers but AMD classifies as human

    **Possible causes:**
    1. Unconventional voicemail greeting without keywords
    2. Very short greeting
    3. Text-based AMD doesn't detect keywords
    4. Poor audio quality affecting ML model

    **Impact:**
    - Agent starts conversation with voicemail system
    - Awkward recording of agent's opening
    - Wasted agent time
    - Unprofessional voicemail

    **Solutions:**
    - Switch to ML-based AMD for better accuracy
    - Add detection in agent instructions:
      ```
      If you say hello and don't receive response within 3 seconds,
      or if you hear typical voicemail sounds (beep, silence, echo),
      switch to voicemail mode and deliver brief message.
      ```
    - Review AMD logs for patterns, report systematic issues
  </Accordion>

  <Accordion title="Agent Speaks Before Beep" icon="volume-xmark">
    **Symptoms:** Voicemail message is partially cut off or not recorded

    **Possible causes:**
    1. Beep detection not working
    2. Agent starts speaking too quickly
    3. Voicemail system doesn't have standard beep

    **Solutions:**
    - Increase wait time for beep in instructions:
      ```
      Wait up to 10 seconds for recording beep before speaking.
      Listen for characteristic tone (usually 1000Hz, 1-2 seconds).
      If no beep after 10 seconds, begin message anyway.
      ```
    - Add explicit pause instruction:
      ```
      After voicemail detected, pause for 3 full seconds before
      beginning message, regardless of beep detection.
      ```
    - Test with various voicemail systems to ensure compatibility
  </Accordion>

  <Accordion title="AMD Takes Too Long" icon="hourglass">
    **Symptoms:** Noticeable delay after call connects before agent responds

    **Possible causes:**
    1. Using ML-based AMD (inherently slower)
    2. Poor audio quality requiring more analysis time
    3. Network latency

    **Solutions:**
    - If speed critical and B2B campaign, switch to text-based AMD
    - Optimize network connection to reduce latency
    - Accept trade-off: Slower AMD = higher accuracy
    - Consider if 1-2 second delay actually matters for use case
  </Accordion>

  <Accordion title="Inconsistent AMD Performance" icon="chart-mixed">
    **Symptoms:** AMD works well sometimes, poorly other times

    **Possible causes:**
    1. Varying call quality (different networks, VoIP vs landline)
    2. Different types of voicemail systems
    3. Time of day affecting answering patterns

    **Solutions:**
    - Review AMD logs for patterns (specific contact types failing)
    - Test both AMD methods on problematic contacts
    - Monitor by contact type (mobile vs landline)
    - Check if certain phone number prefixes have issues
    - May need to segment campaign by phone type and use different AMD methods
  </Accordion>
</AccordionGroup>
