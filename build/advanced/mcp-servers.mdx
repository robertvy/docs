## Overview

Model Context Protocol (MCP) servers transform your AI agents from standalone assistants into powerful platforms connected to external systems, databases, and specialized tools. By attaching MCP servers, you give your agents access to real-time data, computational capabilities, and integrations that go far beyond their base training - enabling them to query databases, fetch live information, execute complex calculations, and interact with third-party services seamlessly during conversations.

MCP represents a standardized way for AI models to connect with external resources, making it easy to extend agent capabilities without custom integrations or complex API development.

<Note>
MCP servers are configured at the account level and then selectively attached to individual agents. The MCP Controls section appears in the agent's Abilities tab for managing server connections.
</Note>

## What is Model Context Protocol?

### Understanding MCP

**Model Context Protocol (MCP)** is an open standard that defines how AI models interact with external tools and data sources in a consistent, secure, and efficient way.

**Key concepts:**

<CardGroup cols={2}>
  <Card title="Protocol Standard" icon="file-contract">
    Standardized communication format between AI models and tools
  </Card>
  <Card title="Server Architecture" icon="server">
    External services expose capabilities via MCP server interface
  </Card>
  <Card title="Tool Discovery" icon="magnifying-glass">
    Agents automatically discover available tools from connected servers
  </Card>
  <Card title="Secure Execution" icon="shield-check">
    Controlled, authenticated access to external resources
  </Card>
</CardGroup>

### How MCP Works

**Traditional approach (before MCP):**
```
Agent needs external data
→ Custom API integration required for each service
→ Different authentication, formats, error handling per integration
→ Maintenance overhead for each custom connection
→ Limited discoverability of available tools
```

**MCP approach:**
```
Agent needs external data
→ Connect to MCP server (standardized interface)
→ Server exposes tools via MCP protocol
→ Agent discovers available tools automatically
→ Agent calls tools using standard MCP format
→ Server handles authentication, execution, and response
→ One protocol, many capabilities
```

**Example flow:**
```
1. Agent connected to "Database MCP Server"
2. Customer asks: "What orders do I have pending?"
3. Agent discovers tool: query_customer_orders
4. Agent calls: query_customer_orders(customer_id="12345")
5. MCP server executes database query
6. Returns: [Order #789, Order #790]
7. Agent responds: "You have 2 pending orders: #789 and #790"
```

### Benefits of MCP

**For agents:**
- **Extended capabilities** - Access to tools beyond base model training
- **Real-time data** - Query current information, not just training data
- **Specialized functions** - Mathematical calculations, data analysis, file operations
- **System integrations** - Connect to databases, APIs, internal systems

**For developers:**
- **Standardization** - One protocol instead of many custom integrations
- **Reusability** - Same MCP server can be used by multiple agents
- **Maintainability** - Update server capabilities without changing agent code
- **Discoverability** - Agents automatically find and use available tools

**For users:**
- **More capable agents** - Agents can do more than just conversation
- **Accurate information** - Real-time data instead of potentially outdated knowledge
- **Seamless experience** - Integrations work naturally in conversation
- **Reliable execution** - Standardized error handling and retries

## Configuring MCP Servers

### Account-Level Setup

MCP servers must first be configured at the account level before attaching to agents:

<Steps>
  <Step title="Access MCP Configuration">
    Navigate to **Account Settings**

    Go to **Integrations** or **MCP Servers** section

    (Exact location may vary - check settings menu)
  </Step>

  <Step title="Provide Smithery Credentials (optional)">
    To browse the full Smithery registry you must set the `SMITHERY_API_KEY` environment variable on your backend (see `/backend/django_app/api/mcp/views.py`). Without it, only the curated Time + Context7 templates appear.
  </Step>

  <Step title="Add New MCP Server">
    Click **Add MCP Server** or **Connect Server**

    Choose from:
    - **MCP Catalog (Smithery)** - Full registry when `SMITHERY_API_KEY` is present, otherwise just the Time/Context7 templates
    - **Custom Server** - Connect your own MCP server
  </Step>

  <Step title="Configure Server Connection">
    **For catalog servers:**
    1. Select server from list
    2. Enter required credentials (API keys, database URLs, etc.)
    3. Configure authentication settings
    4. Test connection
    5. Save server

    **For custom servers:**
    1. Enter server endpoint URL
    2. Configure authentication (Bearer token, API key, etc.)
    3. Specify server capabilities (or auto-discover)
    4. Test connection
    5. Save server
  </Step>

  <Step title="Verify Server Status">
    Server appears in account MCP server list

    Status should show: **Connected** or **Active**

    Tools exposed by server are listed
  </Step>
</Steps>

### Security Considerations

<AccordionGroup>
  <Accordion title="Use Read-Only Credentials When Possible" icon="lock">
    **Principle of least privilege:**

    Create dedicated credentials for MCP servers with minimum necessary permissions

    **Example (Database):**
    ```sql
    -- Bad: Full access
    GRANT ALL PRIVILEGES ON DATABASE customers TO mcp_user;

    -- Good: Read-only access
    CREATE USER mcp_readonly WITH PASSWORD 'secure_password';
    GRANT SELECT ON ALL TABLES IN SCHEMA public TO mcp_readonly;
    ```

    **Why:** If credentials compromised, read-only limits damage
  </Accordion>

  <Accordion title="Rotate Credentials Regularly" icon="rotate">
    **Best practice:**
    - Rotate API keys, passwords every 90 days
    - Update MCP server configuration with new credentials
    - Revoke old credentials immediately after rotation

    **Automation:**
    - Set calendar reminders for credential rotation
    - Use secret management tools (AWS Secrets Manager, HashiCorp Vault)
    - Monitor for approaching expiration dates
  </Accordion>

  <Accordion title="Limit Server Scope" icon="filter">
    **Don't expose entire system:**

    Configure servers to expose only necessary tools

    **Example:**
    ```
    Bad Configuration:
    Salesforce MCP Server with full read/write access to all objects

    Good Configuration:
    Salesforce MCP Server limited to:
    - Read Contacts (phone, email, name only)
    - Read Accounts (company, industry only)
    - No write access
    - No access to sensitive fields (SSN, financial data)
    ```
  </Accordion>

  <Accordion title="Monitor MCP Server Usage" icon="chart-line">
    **Tracking and auditing:**
    - Log all MCP tool executions
    - Monitor for unusual patterns (high volume, unexpected tools)
    - Set alerts for sensitive operations
    - Review logs regularly for security issues

    **Red flags:**
    - Sudden spike in tool usage
    - Queries to sensitive tables
    - Failed authentication attempts
    - Tool calls outside business hours
  </Accordion>
</AccordionGroup>

## Attaching MCP Servers to Agents

### Agent-Level Configuration

Once servers are configured at account level, attach them to specific agents:

<Steps>
  <Step title="Open Agent Configuration">
    Navigate to your agent

    Go to **Abilities** tab

    Scroll to **MCP Servers** section
  </Step>

  <Step title="Attach Server">
    Click **Attach MCP Server** button

    Server selector opens showing available servers

    **Displayed for each server:**
    - Server name
    - Server type/description
    - Number of tools available
    - Connection status
  </Step>

  <Step title="Select Server(s)">
    Choose server(s) to attach to this agent

    Multiple servers can be attached

    Click **Attach** or **Add**
  </Step>

  <Step title="Verify Connection">
    Attached server appears as card in MCP Servers section

    **Card displays:**
    - Server icon and name
    - Description
    - Tool count
    - Remove button
  </Step>

  <Step title="Save Changes">
    Save agent configuration

    Server tools immediately available to agent
  </Step>
</Steps>

### Removing Servers

To disconnect a server from an agent:

1. Find server card in MCP Servers section
2. Click **Remove** button
3. Confirm removal
4. Save agent configuration
5. Tools immediately unavailable to agent

<Warning>
Removing an MCP server from an agent doesn't delete the server from your account. It only disconnects it from this specific agent. Other agents using the same server are unaffected.
</Warning>

## Using MCP Tools in Conversations

### How Agents Use MCP Tools

Once attached, MCP tools work automatically based on conversation context:

**Automatic tool discovery:**
```
Agent receives message: "What orders do I have pending?"

Agent thinks:
1. I need order information for this customer
2. Check available tools...
3. Found: query_customer_orders (from Customer Database MCP Server)
4. This tool can retrieve order data
5. I have customer_id from contact record
6. Execute: query_customer_orders(customer_id="12345", status="pending")
7. Response: [Order #789: Widget A, Order #790: Widget B]
8. Format response: "You have 2 pending orders..."
```

**The agent doesn't need explicit instructions** - it discovers and uses appropriate tools automatically based on:
- Available tool descriptions
- Conversation context
- Customer request
- Contact/campaign data

### Instructing Tool Usage

While agents auto-discover tools, you can guide when and how to use them:

**Example instructions with MCP:**
```
You have access to the Customer Database via MCP tools.

When a customer asks about their orders:
1. Use the query_customer_orders tool with their customer_id
2. Filter by status if they specify (pending, shipped, delivered)
3. Summarize results in a friendly way
4. If no orders found, double-check using get_customer_account tool

When a customer asks about account balance:
1. Use the get_customer_account tool
2. Retrieve the 'balance' field
3. Format as currency
4. Mention any pending charges

Always confirm customer identity before querying sensitive information.
```

**Conditional tool usage:**
```
{% if contact.customer_id %}
You have access to this customer's full order history via MCP tools.
Use query_customer_orders when they ask about orders.
{% else %}
Customer is not yet in our system. Offer to create account or
provide general information only. Do not attempt to query orders.
{% endif %}
```

### Tool Execution Flow

**Real conversation example:**

```
Customer: "Can you tell me when my last order was delivered?"

Agent internal process:
1. Identify intent: Check order delivery date
2. Extract parameters: customer_id from contact record
3. Select tool: query_customer_orders
4. Build query: status="delivered", sort="date_desc", limit=1
5. Execute MCP tool call:

   MCP Request:
   {
     "tool": "query_customer_orders",
     "parameters": {
       "customer_id": "12345",
       "status": "delivered",
       "sort": "date_desc",
       "limit": 1
     }
   }

6. MCP Server processes query:
   → Authenticates with database
   → Executes SQL: SELECT * FROM orders WHERE...
   → Returns results

   MCP Response:
   {
     "orders": [{
       "order_id": "ORD-888",
       "product": "Blue Widget",
       "delivered_date": "2025-11-05",
       "tracking": "1Z999AA1234567890"
     }]
   }

7. Agent formats response:

Agent: "Your last order was delivered on November 5th. That was
       order #888 for the Blue Widget. Would you like the tracking
       number or any other details?"
```

**Error handling:**
```
MCP tool fails (timeout, error, not found):

Agent: "I'm having trouble accessing your order information right
       now. Let me try another way, or I can transfer you to someone
       who can look that up directly. Which would you prefer?"

[Agent follows error handling instructions, doesn't expose technical details]
```

## Best Practices

<AccordionGroup>
  <Accordion title="Start with Read-Only Tools" icon="eye">
    **For initial deployments:**

    Only attach MCP servers with read-only capabilities

    **Why:**
    - Lower risk if agent misbehaves
    - Easier to test and validate
    - Build confidence before enabling write operations

    **Progression:**
    1. **Phase 1:** Read-only tools (queries, lookups, get operations)
    2. **Phase 2:** Low-risk writes (create non-critical records, send notifications)
    3. **Phase 3:** Higher-risk writes (update customer data, financial transactions)

    Test thoroughly at each phase before advancing.
  </Accordion>

  <Accordion title="Provide Clear Tool Descriptions" icon="file-text">
    **When configuring custom MCP servers:**

    Write detailed descriptions for each tool so agents know when and how to use them

    **Bad description:**
    ```
    query_database - Queries database
    ```

    **Good description:**
    ```
    query_customer_orders - Retrieves order history for a specific customer.

    Parameters:
    - customer_id (required): Customer's unique identifier
    - status (optional): Filter by order status (pending, shipped, delivered, cancelled)
    - limit (optional): Maximum number of orders to return (default: 10)

    Returns: List of orders with order_id, product, status, date, and tracking info

    Use when: Customer asks about their orders, delivery status, or order history
    ```

    Clear descriptions help agents select the right tool and use it correctly.
  </Accordion>

  <Accordion title="Monitor MCP Tool Usage" icon="magnifying-glass-chart">
    **Track and analyze how agents use MCP tools:**

    **Metrics to monitor:**
    - Tool execution frequency
    - Success vs. error rate
    - Average execution time
    - Tools never used (potential to remove)
    - Tools heavily used (optimize for performance)

    **Action plan:**
    ```
    Weekly review:
    1. Check MCP tool logs for errors or unusual patterns
    2. Identify underutilized tools → Remove or improve descriptions
    3. Identify overutilized tools → Optimize for performance
    4. Review failed tool calls → Fix data issues or update instructions
    ```
  </Accordion>

  <Accordion title="Handle Tool Failures Gracefully" icon="circle-exclamation">
    **Instruct agents how to handle MCP tool errors:**

    ```
    Instructions example:

    When an MCP tool fails (timeout, error, not found):

    1. Don't expose technical error details to customer
    2. Acknowledge the issue professionally:
       "I'm having trouble accessing that information right now."

    3. Offer alternatives:
       - "Let me try a different approach..."
       - "I can transfer you to someone who can look that up directly."
       - "If you provide [specific information], I can help another way."

    4. If tool repeatedly fails, escalate:
       - Transfer to human agent
       - Create support ticket for manual follow-up

    Never say: "MCP query_database tool returned error 500"
    Instead say: "I'm having trouble retrieving that information right now.
                 Let me connect you with someone who can help directly."
    ```
  </Accordion>

  <Accordion title="Test Tools Before Production" icon="vial">
    **Comprehensive testing strategy:**

    <Steps>
      <Step title="Test in Isolation">
        Test each MCP tool individually using test interface or API client

## Troubleshooting

<AccordionGroup>
  <Accordion title="MCP Server Not Appearing in Agent Selector" icon="eye-slash">
    **Symptoms:** Server configured at account level but not available when attaching to agent

    **Possible causes:**
    1. Server not saved/activated at account level
    2. Server connection failed
    3. Browser cache issue
    4. Permissions issue

    **Solutions:**
    - Go to account MCP servers, verify server shows as "Connected"
    - Click "Test Connection" to verify server is reachable
    - Refresh browser / clear cache
    - Check account permissions (admin role may be required)
    - Try different browser
  </Accordion>

  <Accordion title="Agent Not Using MCP Tools" icon="robot">
    **Symptoms:** Tools are attached but agent doesn't use them in conversations

    **Possible causes:**
    1. Tool descriptions unclear
    2. Agent doesn't recognize when to use tool
    3. Required parameters not available
    4. Instructions conflicting with tool usage

    **Solutions:**
    - **Improve tool descriptions:** Make them more explicit about when to use
    - **Add instructions:** Guide agent on tool usage:
      ```
      When customer asks about orders, always use query_customer_orders
      tool to get accurate information. Don't rely on general knowledge.
      ```
    - **Check parameters:** Ensure agent has access to required data (customer_id, etc.)
    - **Test explicitly:** Ask questions that clearly require tool usage
  </Accordion>

  <Accordion title="Tool Execution Errors" icon="triangle-exclamation">
    **Symptoms:** Tools fail when agent tries to use them

    **Possible causes:**
    1. Authentication failure (expired credentials)
    2. Network connectivity issue
    3. External service down
    4. Invalid parameters passed by agent
    5. Permissions insufficient

    **Solutions:**
    - **Check credentials:** Verify API keys, passwords haven't expired
    - **Test connection:** Use "Test Connection" button in server config
    - **Review logs:** Check MCP server logs for specific error messages
    - **Verify permissions:** Ensure server credentials have necessary access
    - **Check external service:** Confirm third-party service is operational
    - **Validate parameters:** Review tool call logs to see what agent is passing
  </Accordion>

  <Accordion title="Slow Tool Performance" icon="hourglass">
    **Symptoms:** MCP tools take very long to execute (>5 seconds)

    **Possible causes:**
    1. External service slow (database, API)
    2. Complex queries inefficient
    3. Network latency
    4. Connection pool exhausted
    5. Large data transfers

    **Solutions:**
    - **Optimize queries:** Add database indexes, simplify queries
    - **Increase connection pool:** Allow more concurrent connections
    - **Reduce data transfer:** Return only necessary fields, add pagination
    - **Cache results:** For frequently accessed data, implement caching
    - **Monitor infrastructure:** Check database/API server performance
    - **Consider async:** For very slow operations, use webhooks instead
  </Accordion>

  <Accordion title="Incorrect Results from Tools" icon="circle-xmark">
    **Symptoms:** Tool returns data but it's wrong or unexpected

    **Possible causes:**
    1. Agent passing wrong parameters
    2. Tool logic incorrect (custom servers)
    3. Data quality issue in source system
    4. Mismatched data formats

    **Solutions:**
    - **Review tool call logs:** Check parameters agent is passing
    - **Test tool directly:** Call tool with known parameters, verify results
    - **Check source data:** Query database/API directly to confirm data
    - **Update tool logic:** Fix bugs in custom MCP server implementation
    - **Add validation:** Validate parameters before executing tool
    - **Update instructions:** Guide agent to pass correct parameters
  </Accordion>
</AccordionGroup>
