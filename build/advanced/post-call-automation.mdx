## Overview

Post-call email automation sends targeted follow-up emails based on conversation outcomes. Emails trigger automatically when calls meet specific criteria like goal achievement, sentiment analysis, or custom conditions you define.

**Access:** Configure in agent editor under **Analytics → Post-Call Emails**

---

## How It Works

<Steps>
  <Step title="Configure trigger conditions">
    Define when emails should send:
    - Goal achieved/not achieved
    - Specific analysis outputs
    - Tags matched during conversation
    - Custom criteria combinations

    <Info>
    Provide a detailed description when configuring triggers. The AI analyzes this description against the call transcript to determine if conditions are met.
    </Info>
  </Step>

  <Step title="Design email template">
    Create email with:
    - Subject line with variables
    - HTML or plain text body
    - Dynamic content from conversation
    - System and custom variables
  </Step>

  <Step title="Enable automation">
    Turn on the notification

    Test with preview feature before going live
  </Step>

  <Step title="Emails send automatically">
    After each matching call:
    - System checks conditions
    - If met, renders email template
    - Sends to customer's email address
  </Step>
</Steps>

## Creating Email Automations

### Step 1: Access Post-Call Emails

1. Open your agent in the editor
2. Go to **Analytics → Post-Call Emails**
3. Click **Create Email** or choose from templates

### Step 2: Configure Trigger Conditions

Choose when this email should send:

<Tip>
**Important:** Write detailed, specific trigger descriptions. The AI analyzes your description against the full call transcript to determine if conditions are met. More detailed descriptions = more accurate triggering.
</Tip>

**Goal-based triggers:**
```
Send when: "Book Appointment" goal is achieved
```

**Analysis-based triggers:**
```
Send when: Sentiment = "Negative" AND Issue = "Unresolved"

Description: "Did the customer express frustration, dissatisfaction, or
negative sentiment, AND did the conversation end without fully resolving
their issue or answering their question?"
```

**Tag-based triggers:**
```
Send when: Conversation tagged with "follow-up-needed"

Description: "Did the agent promise to follow up, send information later,
or tell the customer someone would contact them?"
```

**Combined conditions:**
```
Send when:
- Goal "Collect Information" achieved
- AND contact.email is present
- AND conversation_duration > 60 seconds

Description: "Did the agent successfully collect the required information
from the customer (name, email, phone, reason for inquiry), and did the
customer provide an email address?"
```

### Step 3: Design Your Email

**Subject line example:**
```
Thanks for calling {{agent_name}}, {{customer_name}}!
```

**Email body example:**
```html
Hi {{customer_name}},

Thanks for speaking with us today. Here's a summary of our conversation:

{{conversation_summary}}

Your appointment is confirmed for {{dyn_appointment_date}} at {{dyn_appointment_time}}.

Meeting link: {{dyn_meeting_link}}

If you need to reschedule, reply to this email or call us at {{agent_number}}.

Best regards,
{{agent_name}}
```

### Step 4: Preview and Test

1. Click **Preview** to see rendered output
2. System shows example with sample data
3. Verify formatting and variable replacement
4. Make adjustments if needed

### Step 5: Enable

Toggle the notification **On** to activate it.

## Template Variables

### System Variables

Always available from conversation and contact data:

**Conversation Details:**
- `{{conversation_date}}` - Date of call (e.g., "2025-11-12")
- `{{conversation_time}}` - Time of call (e.g., "14:30")
- `{{conversation_duration}}` - Length in seconds
- `{{conversation_summary}}` - AI-generated summary
- `{{conversation_url}}` - Link to view in dashboard
- `{{conversation_status}}` - ended, transferred, etc.

**Contact Information:**
- `{{customer_name}}` - Contact's full name
- `{{customer_first_name}}` - Contact's first name
- `{{customer_email}}` - Contact's email address
- `{{customer_number}}` - Contact's phone number
- `{{contact.company}}` - Company name (if available)

**Agent & Account:**
- `{{agent_name}}` - Name of the AI agent
- `{{agent_number}}` - Agent's phone number
- `{{account_name}}` - Your company name

### Dynamic Variables

Extract custom information from transcripts using the `dyn_` prefix:

**How dynamic variables work:**
1. You reference `{{dyn_issue_summary}}` in your email
2. System uses conversation transcript + post-call analysis
3. AI extracts the specific information requested
4. Populates the variable in the email

**Example dynamic variables:**
- `{{dyn_issue_summary}}` - AI extracts issue description
- `{{dyn_action_items}}` - AI lists next steps discussed
- `{{dyn_appointment_date}}` - AI extracts mentioned date
- `{{dyn_ticket_number}}` - AI finds ticket reference
- `{{dyn_product_interest}}` - AI identifies product discussed
- `{{dyn_followup_owner}}` - AI determines who should follow up

**Naming rules:**
- Must start with `dyn_`
- Only letters, numbers, underscores
- No spaces, hyphens, or special characters
- Examples: `dyn_order_id`, `dyn_next_meeting`, `dyn_issue_type`

<Info>
**Tip:** Make dynamic variable names descriptive. The clearer the name, the better the AI understands what to extract. Use `{{dyn_resolved_issue_summary}}` instead of `{{dyn_info}}`.
</Info>

## Managing Email Automations

### Edit Existing Email

1. Go to **Analytics → Post-Call Emails**
2. Click the email automation you want to edit
3. Modify conditions, subject, or body
4. Click **Preview** to test changes
5. **Save** to apply updates

### Duplicate Notification

Create variants for A/B testing:
1. Click menu (⋮) next to notification
2. Select **Duplicate**
3. Modify subject or body
4. Enable one version at a time to test

### Disable Temporarily

Keep configuration but stop sending:
1. Toggle notification to **Off**
2. Notification saved but won't trigger
3. Toggle back **On** when ready

### Delete Notification

Permanently remove:
1. Click menu (⋮) next to notification
2. Select **Delete**
3. Confirm deletion

## Best Practices

<AccordionGroup>
  <Accordion title="Start Simple" icon="seedling">
    Begin with basic system variables before adding dynamic variables.

    **Simple first version:**
    ```
    Hi {{customer_name}},

    Thanks for calling us today.

    Conversation summary:
    {{conversation_summary}}

    View full details: {{conversation_url}}
    ```

    **Add dynamic variables after testing:**
    ```
    Hi {{customer_name}},

    Issue discussed: {{dyn_issue_type}}
    Resolution: {{dyn_resolution_steps}}
    Ticket: {{dyn_ticket_number}}
    ```
  </Accordion>

  <Accordion title="Test Before Enabling" icon="vial">
    Always use the **Preview** feature:

    1. Click **Preview** in notification editor
    2. Review rendered email with sample data
    3. Check for:
       - Formatting issues
       - Missing variables
       - Unclear dynamic variables
    4. Make adjustments
    5. Preview again
    6. Only then enable
  </Accordion>

  <Accordion title="Verify Email Addresses" icon="envelope-circle-check">
    Ensure contacts have valid emails:

    **Priority order** (system checks in this order):
    1. Email from context API (`cal_email` or custom field)
    2. Email stored in contact record
    3. Email collected during conversation

    **In agent instructions:**
    ```
    If the customer needs a follow-up email:
    1. Check if we have their email on file
    2. If not, ask: "What email should I send this to?"
    3. Confirm the email address is correct
    ```
  </Accordion>

  <Accordion title="Make Dynamic Variables Descriptive" icon="tag">
    Clear variable names help AI extract correctly:

    ✅ **Good:**
    - `{{dyn_appointment_date_and_time}}`
    - `{{dyn_unresolved_issue_description}}`
    - `{{dyn_recommended_next_steps}}`

    ❌ **Bad:**
    - `{{dyn_info}}`
    - `{{dyn_data}}`
    - `{{dyn_thing}}`

    The variable name itself guides the AI on what to extract.
  </Accordion>

  <Accordion title="Keep Emails Concise" icon="align-left">
    Shorter emails have higher engagement:

    - One clear purpose per email
    - 3-5 short paragraphs maximum
    - Bullet points for multiple items
    - Single clear call-to-action

    **Example structure:**
    1. Greeting with name
    2. Reference to conversation (1-2 lines)
    3. Key information (3-5 bullets)
    4. Next steps (1-2 actions)
    5. Contact info/signature
  </Accordion>

  <Accordion title="Handle Missing Data Gracefully" icon="circle-question">
    Dynamic variables might not always extract successfully.

    **Fallback patterns:**

    Instead of:
    ```
    Your appointment: {{dyn_appointment_date}}
    ```

    Use:
    ```
    {% if dyn_appointment_date %}
    Your appointment: {{dyn_appointment_date}}
    {% else %}
    We discussed scheduling an appointment. I'll follow up shortly to confirm the date.
    {% endif %}
    ```

    **Note:** Check if conditional logic is supported in your template system. If not, ensure critical info is captured in system variables or contact fields.
  </Accordion>

  <Accordion title="Monitor Delivery" icon="chart-line">
    Track email automation performance:

    - Check conversation details for sent emails
    - Monitor bounce rates
    - Review customer replies
    - Test periodically with your own email
    - Update templates based on engagement
  </Accordion>
</AccordionGroup>

## Troubleshooting

<AccordionGroup>
  <Accordion title="Email Not Sending" icon="envelope-open-text">
    **Possible causes:**
    - Notification is disabled (toggle is Off)
    - Trigger conditions not met
    - Customer has no email address
    - Email address is invalid

    **Solution:**
    1. Verify notification is enabled (toggle On)
    2. Check conversation matches trigger conditions
    3. Verify contact has valid email in system
    4. Review conversation logs for errors
  </Accordion>

  <Accordion title="Variables Not Populating" icon="brackets-curly">
    **Possible causes:**
    - Typo in variable name
    - Variable doesn't exist in system
    - Dynamic variable too vague
    - Data not available in conversation

    **Solution:**
    - Use exact variable names from picker
    - Verify variable is supported
    - Make dynamic variables more descriptive
    - Use **Preview** to test before enabling
  </Accordion>

  <Accordion title="Dynamic Variables Extracting Wrong Data" icon="robot">
    **Possible causes:**
    - Variable name not clear enough
    - Multiple similar pieces of information in transcript
    - Conversation ambiguous about the data

    **Solution:**
    - Make variable names more specific:
      - `{{dyn_next_appointment_date}}` not `{{dyn_date}}`
      - `{{dyn_shipping_address}}` not `{{dyn_address}}`
    - Add context in post-call analysis configuration
    - Have agent confirm details during call
  </Accordion>

  <Accordion title="Emails Going to Spam" icon="shield-exclamation">
    **Possible causes:**
    - No SPF/DKIM configured for sending domain
    - Too many links in email
    - Spam-trigger words in subject/body

    **Solution:**
    - Contact support to verify email authentication
    - Reduce number of links
    - Avoid all-caps, excessive punctuation
    - Test with multiple email providers
  </Accordion>

  <Accordion title="Customer Didn't Receive Email" icon="inbox">
    **Quick checks:**
    1. Check conversation details - was email sent?
    2. Verify customer's email address is correct
    3. Ask customer to check spam/junk folder
    4. Verify notification was enabled at time of call
    5. Check trigger conditions were met
  </Accordion>
</AccordionGroup>

## Advanced: Linking to Post-Call Analysis

Dynamic variables become even more powerful when combined with post-call analysis.

**Setup flow:**

<Steps>
  <Step title="Configure post-call analysis">
    In **Analytics → Call Analysis**, define structured questions:

    ```json
    {
      "issue_category": "string",
      "resolution_status": "resolved | pending | escalated",
      "action_items": "array of strings",
      "followup_needed": "boolean"
    }
    ```
  </Step>

  <Step title="Reference analysis in email">
    Use analysis results directly:

    ```
    Issue Category: {{analytics.issue_category}}
    Status: {{analytics.resolution_status}}

    Next steps:
    {{analytics.action_items}}
    ```
  </Step>

  <Step title="Trigger based on analysis">
    Set conditions using analysis outputs:

    ```
    Send when: analytics.followup_needed = true
    ```
  </Step>
</Steps>

<Info>
This creates a tight feedback loop: analysis identifies patterns → emails automate follow-up → you optimize based on results.
</Info>
