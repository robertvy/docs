## Overview

Custom API actions transform your AI agents from conversation-only assistants into comprehensive business automation tools that integrate seamlessly with your existing systems. By connecting to external APIs via HTTP, your agents can retrieve customer data, update CRM records, process transactions, trigger workflows, and execute business logic - all during natural conversations.

This is where the real power of AI agents emerges: combining conversational AI with your business systems to create truly autonomous customer experiences.

<Note>
Custom actions use standard HTTP protocols, making them compatible with virtually any modern system that exposes an API - CRMs, help desks, databases, payment processors, inventory systems, and more.
</Note>

---

## What Are Custom API Actions?

Custom API actions are HTTP-based integrations that allow your agents to communicate with external systems during conversations. When a customer asks a question or makes a request, your agent can:

1. **Call your API** with relevant data (customer info, conversation context)
2. **Receive response** from your system (customer records, inventory status, etc.)
3. **Use that data** to continue the conversation intelligently
4. **Update systems** based on customer input or conversation outcomes

### Real-World Example Flow

```text wrap
Customer: "What's the status of my order?"

Agent: "I'd be happy to check that for you."

[Agent calls 'lookup_order' custom action]
→ GET https://api.company.com/orders?phone=+14155551234

[API responds with order data]
← {
    "order_id": "ORD-12345",
    "status": "shipped",
    "tracking": "1Z999AA1234567890",
    "est_delivery": "2025-01-18"
  }

Agent: "Your order #12345 has shipped! It's on the way and should
       arrive by January 18th. Your tracking number is
       1Z999AA1234567890. Would you like me to send that to your email?"

Customer: "Yes please"

[Agent calls 'send_tracking_email' custom action]
→ POST https://api.company.com/notifications/email
  Body: {"email": "customer@example.com", "tracking": "1Z999..."}

Agent: "Done! I've sent the tracking information to your email."
```text wrap

This entire interaction happens in seconds during a live conversation, with no human intervention required.

<CodeGroup>
```bash lookup-order.sh theme={null}
curl --request GET \
  --url "https://api.company.com/orders" \
  --header "Authorization: Bearer $API_TOKEN" \
  --data-urlencode "phone={{ contact.phone_number }}"
```text wrap

```javascript sendTracking.js theme={null}
await fetch("https://api.company.com/notifications/email", {
  method: "POST",
  headers: {
    Authorization: `Bearer ${env.API_TOKEN}`,
    "content-type": "application/json"
  },
  body: JSON.stringify({
    email: contact.email,
    tracking: context.order.tracking
  })
});
```text wrap
</CodeGroup>

## HTTP Method Support

Custom actions support all standard HTTP methods for different types of operations:

<CardGroup cols={2}>
  <Card title="GET" icon="download">
    Retrieve information from external systems without modifying data
  </Card>
  <Card title="POST" icon="paper-plane">
    Create new records or submit data to external systems
  </Card>
  <Card title="PUT" icon="pen-to-square">
    Update complete records with full replacement
  </Card>
  <Card title="PATCH" icon="pen">
    Modify specific fields in existing records
  </Card>
  <Card title="DELETE" icon="trash">
    Remove records from external systems
  </Card>
</CardGroup>

### When to Use Each Method

**GET - Retrieve Data:**
```text wrap
Use cases:
- Look up customer account information
- Check order status
- Retrieve product inventory
- Get pricing information
- Fetch account balance

Example: GET https://crm.company.com/customers/12345
```text wrap

**POST - Create New Data:**
```text wrap
Use cases:
- Create support tickets
- Submit lead information
- Log conversation notes
- Send notifications
- Trigger webhooks

Example: POST https://helpdesk.company.com/tickets
```text wrap

**PUT - Replace Data:**
```text wrap
Use cases:
- Update entire customer record
- Replace configuration settings
- Overwrite document content

Example: PUT https://crm.company.com/customers/12345
```text wrap

**PATCH - Modify Data:**
```text wrap
Use cases:
- Update lead status
- Change customer preferences
- Modify specific fields
- Increment counters

Example: PATCH https://crm.company.com/leads/789
```text wrap

**DELETE - Remove Data:**
```text wrap
Use cases:
- Cancel appointments
- Remove items from cart
- Delete temporary records
- Unsubscribe from lists

Example: DELETE https://api.company.com/subscriptions/456
```text wrap

<Warning>
DELETE operations are permanent. Ensure your agents only use DELETE actions when explicitly instructed and after proper confirmation from customers.
</Warning>

## Creating a Custom Action

<Steps>
  <Step title="Navigate to Actions">
    Open your agent and go to **Abilities → Actions** tab
  </Step>
  <Step title="Add Custom Action">
    Click **Add Action** and select **Custom API**
  </Step>
  <Step title="Name Your Action">
    Use a descriptive name like "Lookup Customer Account" or "Create Support Ticket"

    This exact name will be referenced in your agent's instructions
  </Step>
  <Step title="Add Description">
    Document what the action does and when to use it:

    "Retrieves customer data from Salesforce using phone number. Use before answering account-specific questions."
  </Step>
  <Step title="Configure HTTP Method">
    Select the appropriate HTTP method (GET, POST, PUT, PATCH, DELETE)
  </Step>
  <Step title="Set Endpoint URL">
    Enter your API endpoint with optional template variables:

    `https://api.crm.com/customers/{{ contact.phone_number }}`
  </Step>
  <Step title="Add Headers">
    Configure required HTTP headers (Content-Type, API version, etc.)
  </Step>
  <Step title="Configure Authentication">
    Set up authentication using one of the supported methods (detailed below)
  </Step>
  <Step title="Add Parameters or Body">
    For GET: Add query parameters
    For POST/PUT/PATCH: Configure request body
  </Step>
  <Step title="Save and Test">
    Save the action and test it in the agent test interface
  </Step>
</Steps>

<Columns cols={2}>
  <Card title="API key header" icon="key">
    Simplest option. Store the key in **Core → Secrets** and inject it as a static header.
  </Card>

  <Card title="OAuth 2.0 client credentials" icon="lock">
    For services that issue short-lived tokens. Create a pre-action hook that exchanges client ID/secret for an access token and pass it to your action.
  </Card>
</Columns>

## Authentication Methods

Custom actions support five authentication schemes:

### None - No Authentication

Use for public APIs that don't require authentication.

```text wrap


 Auth Type: NONE  

 No authentication required for this endpoint  


```text wrap

**Use cases:**
- Public APIs
- Internal endpoints on private network
- APIs using other security (IP whitelisting, VPN)

<Warning>
Only use NONE authentication for truly public endpoints. Most production APIs should use authentication to prevent unauthorized access.
</Warning>

### Bearer Token Authentication

Most common for modern APIs. Sends a token in the Authorization header.

```text wrap


 Auth Type: BEARER  

 Bearer Token*  
 [eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9.eyJzdWIiOiIx•••••]  

 ℹ️  Token will be sent as: Authorization: Bearer {token}  


```text wrap

**HTTP header sent:**
```text wrap
Authorization: Bearer eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...
```text wrap

**Use cases:**
- JWT authentication
- OAuth 2.0 access tokens
- API keys in bearer format
- Modern REST APIs

**Configuration example:**
```json wrap
{
  "auth_settings": {
    "auth_type": "BEARER",
    "auth_value": {
      "value": "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9..."
    }
  }
}
```text wrap

### Basic Authentication

Traditional username/password authentication using Base64 encoding.

```text wrap


 Auth Type: BASIC  

 Username*  
 [api_user_________________________________]  

 Password*  
 [••••••••••••••••••••••••••••••••••••]  

 ℹ️  Credentials will be Base64 encoded and sent in  
    Authorization header  


```text wrap

**HTTP header sent:**
```text wrap
Authorization: Basic YXBpX3VzZXI6cGFzc3dvcmQ=
```text wrap

**Use cases:**
- Legacy APIs
- Internal services
- Simple authentication needs

**Configuration example:**
```json wrap
{
  "auth_settings": {
    "auth_type": "BASIC",
    "auth_value": {
      "username": "api_user",
      "password": "secure_password_here"
    }
  }
}
```text wrap

<Note>
Basic auth should only be used over HTTPS to prevent credential exposure. Most modern APIs prefer Bearer token authentication.
</Note>

### Header-Based Authentication

Custom authentication via HTTP headers (e.g., X-API-Key).

```text wrap


 Auth Type: HEADER  

 Header Name*  
 [X-API-Key_________________________________]  

 Header Value*  
 [sk-abc123def456ghi789•••••••••••••••••]  

 ℹ️  Custom header will be added to all requests  


```text wrap

**HTTP header sent:**
```text wrap
X-API-Key: sk-abc123def456ghi789jkl012mno345pqr678
```text wrap

**Use cases:**
- API key authentication
- Custom auth schemes
- Vendor-specific authentication
- Legacy system integration

**Configuration example:**
```json wrap
{
  "auth_settings": {
    "auth_type": "HEADER",
    "auth_value": {
      "name": "X-API-Key",
      "value": "sk-abc123def456ghi789jkl012mno345pqr678"
    }
  }
}
```text wrap

**Common header names:**
- `X-API-Key`
- `X-Auth-Token`
- `X-Access-Token`
- `Api-Key`
- `Authorization` (custom format)

### Body-Based Authentication

Credentials included in the request body (typically for POST requests).

```text wrap


 Auth Type: BODY  

 Parameter Name*  
 [api_key___________________________________]  

 Parameter Value*  
 [sk-xyz789abc123•••••••••••••••••••••]  

 ℹ️  Authentication parameter will be included in request  
    body JSON  


```text wrap

**Request body includes:**
```json wrap
{
  "api_key": "sk-xyz789abc123",
  "...other parameters..."
}
```text wrap

**Use cases:**
- Login/authentication endpoints
- Systems requiring credentials in body
- Non-standard auth schemes

**Configuration example:**
```json wrap
{
  "auth_settings": {
    "auth_type": "BODY",
    "auth_value": {
      "name": "api_key",
      "value": "sk-xyz789abc123def456"
    }
  }
}
```text wrap

## Request Configuration

### Query Parameters (GET Requests)

For GET requests, add URL query parameters:

```text wrap


 Parameter Name          Value                   [+ Add]  

  include          [orders,preferences_________]  
  limit            [100____________________]  
  format           [json___________________]  


 Resulting URL:  
 https://api.crm.com/customers/123?include=orders,  
 preferences&limit=100&format=json  


```text wrap

**Configuration example:**
```json wrap
{
  "http_method": "GET",
  "endpoint_url": "https://api.crm.com/customers/{{contact.phone_number}}",
  "query_parameters": {
    "include": "orders,preferences",
    "limit": "100",
    "format": "json"
  }
}
```text wrap

### Request Body (POST/PUT/PATCH)

For methods that accept body data, configure the JSON payload:

```text wrap


 Body Content (JSON)  

 {  
   "customer_id": "{{ contact.id }}",  
   "phone": "{{ contact.phone_number }}",  
   "email": "{{ contact.email }}",  
   "call_time": "{{ current_datetime }}",  
   "agent": "{{ agent_name }}",  
   "status": "contacted"  
 }  


 <Icon icon="check-circle" /> Valid JSON   <Icon icon="lightbulb" /> Variables supported  


```text wrap

**Configuration example:**
```json wrap
{
  "http_method": "POST",
  "endpoint_url": "https://crm.company.com/leads",
  "request_body": {
    "customer_id": "{{ contact.id }}",
    "phone": "{{ contact.phone_number }}",
    "email": "{{ contact.email }}",
    "call_time": "{{ current_datetime }}",
    "status": "contacted"
  }
}
```text wrap

<Tip>
Use template variables in your request body to include dynamic data from the conversation, contact record, or campaign context.
</Tip>

## Template Variables

Custom actions support Jinja2 template syntax for dynamic content:

### Available Variables

**Contact Information:**
```text wrap
{{ contact.first_name }}        - Customer's first name
{{ contact.last_name }}         - Customer's last name
{{ contact.email }}             - Email address
{{ contact.phone_number }}      - Phone number (E.164 format)
{{ contact.id }}                - Unique contact ID
{{ contact.custom_field_name }} - Custom contact fields
```text wrap

**Agent Context:**
```text wrap
{{ agent_number }}              - Agent's phone number
{{ agent_name }}                - Agent's display name
{{ agent_id }}                  - Unique agent ID
{{ team }}                      - Team assignment
```text wrap

**Call Details:**
```text wrap
{{ direction }}                 - Call direction (inbound/outbound)
{{ medium }}                    - Communication channel (phone/web)
{{ current_datetime }}          - Current timestamp (ISO 8601)
{{ session_id }}                - Unique session identifier
```text wrap

### Using Variables in URLs

Variables can be used directly in endpoint URLs:

```text wrap
GET https://api.crm.com/customers/{{ contact.phone_number }}

POST https://api.company.com/tickets?agent={{ agent_id }}

PATCH https://api.service.com/leads/{{ contact.id }}/status
```text wrap

### Using Variables in Headers

```json wrap
{
  "request_headers": {
    "Content-Type": "application/json",
    "X-Customer-ID": "{{ contact.id }}",
    "X-Agent-Name": "{{ agent_name }}"
  }
}
```text wrap

### Using Variables in Request Bodies

```json wrap
{
  "request_body": {
    "customer": {
      "first_name": "{{ contact.first_name }}",
      "last_name": "{{ contact.last_name }}",
      "email": "{{ contact.email | default('unknown@example.com') }}",
      "phone": "{{ contact.phone_number }}"
    },
    "call_metadata": {
      "timestamp": "{{ current_datetime }}",
      "direction": "{{ direction }}",
      "agent": "{{ agent_name }}"
    }
  }
}
```text wrap

### Default Values and Filters

Use Jinja filters to handle missing data:

```text wrap
{{ contact.email | default("not-provided@example.com") }}
{{ contact.first_name | default("Customer") }}
{{ contact.phone_number | default("unknown") }}
```text wrap

**Common filters:**
- `default("value")` - Provide fallback if variable is empty
- `upper` - Convert to uppercase
- `lower` - Convert to lowercase
- `title` - Title case
- `trim` - Remove whitespace

## Before-Call vs During-Call Data

Understanding data availability timing is crucial for reliable custom actions:

### Before-Call Data

Available before the conversation starts (from contact record, campaign, etc.):

**Contact record fields:**
- Name, email, phone number
- Custom contact fields
- Account information
- Tags and segments

**Call context:**
- Call direction (inbound/outbound)
- Custom context variables
- Scheduled call time

**Use for:**
- Looking up customer data at call start
- Pre-loading account information
- Routing decisions
- Personalization before greeting

**Example:**
```text wrap
Action: Lookup Customer Account (runs at call start)

Endpoint: GET https://crm.com/customers/{{ contact.phone_number }}

Agent instructions:
Before greeting the customer, I automatically look up their
account information so I can personalize the conversation.
```text wrap

### During-Call Data

Collected or modified during the active conversation:

**Conversation variables:**
- Information the customer provides
- Results from previous actions
- Answers to questions
- Customer selections

**Dynamic context:**
- Current call status
- Time elapsed
- Actions executed
- Decisions made

**Use for:**
- Submitting information collected during call
- Updating records based on conversation
- Triggering workflows based on outcomes
- Creating tickets or leads with call context

**Example:**
```text wrap
Action: Create Support Ticket (runs during call)

Endpoint: POST https://helpdesk.com/tickets

Body:
{
  "customer_email": "{{ contact.email }}",
  "issue_description": "[Collected during conversation]",
  "priority": "[Determined by agent based on conversation]",
  "created_by_agent": "{{ agent_id }}"
}

Agent instructions:
After collecting the customer's issue description and
determining priority, use the 'Create Support Ticket' action
to log the issue in our help desk system.
```text wrap

<Note>
Before-call actions execute automatically when the call starts. During-call actions execute when explicitly invoked by the agent based on conversation context and your instructions.
</Note>

## Response Processing

### How Agents Use API Responses

When a custom action executes, the response is made available to the agent:

1. **API returns data** in JSON format
2. **Platform parses response** automatically
3. **Agent receives structured data** to use in conversation
4. **Agent references fields** naturally in responses

### Example Response Handling

**API returns:**
```json wrap
{
  "customer_id": "cust_12345",
  "name": "John Smith",
  "account_type": "premium",
  "balance": 1250.00,
  "recent_orders": [
    {
      "order_id": "ORD-789",
      "status": "shipped",
      "tracking": "1Z999AA1234567890"
    }
  ]
}
```text wrap

**Agent uses data:**
```text wrap
Agent: "Hi John! I can see you're a premium member with us.
       Your recent order #789 has shipped and should arrive soon.
       Your tracking number is 1Z999AA1234567890."
```text wrap

### Chaining Actions

Use data from one action in subsequent actions:

```text wrap
1. Call 'Lookup Customer' action
   → Returns customer_id: "cust_12345"

2. Call 'Get Order History' action
   → Uses customer_id from step 1
   → Returns list of orders

3. Call 'Send Email' action
   → Uses customer email from step 1
   → Includes order info from step 2
```text wrap

## Error Handling & Failovers

### HTTP Error Codes

Your custom actions should handle common HTTP errors:

**4xx Client Errors:**
- **400 Bad Request** - Invalid request format
- **401 Unauthorized** - Authentication failed
- **403 Forbidden** - Permission denied
- **404 Not Found** - Resource doesn't exist
- **429 Too Many Requests** - Rate limit exceeded

**5xx Server Errors:**
- **500 Internal Server Error** - API server issue
- **502 Bad Gateway** - Upstream server problem
- **503 Service Unavailable** - Temporary outage
- **504 Gateway Timeout** - Request took too long

### Configuring Fallback Behavior

Instruct your agent how to handle failures:

```text wrap
If the 'Lookup Customer' action fails:

1. Check error type:
   - 404 (not found): "I don't see an account with that information"
   - 401/403 (auth): "I'm having trouble accessing the system right now"
   - 500/503 (server): "Our system is temporarily unavailable"
   - Timeout: "The lookup is taking longer than expected"

2. Offer alternatives:
   - "I can have someone call you back with that information"
   - "Let me transfer you to someone who can help"
   - "Could you provide your account number instead?"

3. Don't stop the conversation - continue helping within your capabilities

4. If critical data needed, use fallback:
   - Transfer to human agent
   - Schedule callback
   - Collect information for manual follow-up
```text wrap

### Retry Logic

For transient failures, configure retry behavior:

```text wrap
If action fails with timeout or 5xx error:
1. Wait 2 seconds
2. Retry once
3. If still fails, follow error handling instructions

Don't retry for:
- 4xx errors (client errors - won't succeed on retry)
- 401/403 (authentication - needs credentials fixed)
- 404 (not found - resource doesn't exist)
```text wrap

<Warning>
Be cautious with automatic retries for POST/PUT/PATCH/DELETE actions. Retrying could create duplicate records or unintended changes. Retry logic is safest for GET requests.
</Warning>

### Timeout Configuration

Set appropriate timeouts to prevent long pauses in conversation:

**Recommended timeouts:**
- Simple lookups: 3-5 seconds
- Complex queries: 5-8 seconds
- Write operations: 5-10 seconds
- External integrations: 8-10 seconds

If your API regularly exceeds these timeouts, consider:
- Optimizing API performance
- Using asynchronous webhooks instead
- Pre-loading data before call starts
- Caching frequently accessed data

## Webhooks vs Custom Actions

Understanding the difference helps you choose the right integration method:

### Custom Actions (Synchronous)

**How they work:**
1. Agent decides action is needed during conversation
2. Action executes immediately
3. Agent waits for response
4. Agent uses response to continue conversation

**Best for:**
- Real-time data lookups
- Information needed to continue conversation
- User-initiated operations
- Quick API responses (under 3 seconds)

**Examples:**
- Looking up customer account
- Checking order status
- Verifying inventory
- Getting pricing information

### Webhooks (Asynchronous)

**How they work:**
1. Event occurs (call starts, ends, action executes)
2. Platform sends webhook to your server
3. Your server processes in background
4. No impact on conversation flow

**Best for:**
- Post-call processing
- Analytics and logging
- Slow operations (>3 seconds)
- Operations that don't affect conversation

**Examples:**
- Logging call transcripts
- Updating CRM after call ends
- Triggering follow-up workflows
- Sending data to analytics platforms

<Tip>
Use custom actions for synchronous, conversation-critical operations. Use webhooks for asynchronous, post-conversation processing. Often you'll use both together.
</Tip>

## Security Best Practices

<AccordionGroup>
  <Accordion title="Use HTTPS Only" icon="lock">
    Always use HTTPS endpoints, never HTTP:

    <Icon icon="check" /> `https://api.company.com/endpoint`
    <Icon icon="xmark" /> `http://api.company.com/endpoint`

    HTTP transmits data in plain text, exposing sensitive information.
  </Accordion>

  <Accordion title="Secure Credential Storage" icon="key">
    - Never hardcode credentials in endpoint URLs
    - Use authentication configuration, not URL parameters
    - Rotate API keys regularly
    - Use different keys for testing vs production
    - Revoke compromised credentials immediately
  </Accordion>

  <Accordion title="Implement Proper Authentication" icon="shield-check">
    - Use the strongest authentication method your API supports
    - Prefer Bearer tokens or API keys over Basic auth
    - Use separate API credentials for itellicoAI (easier to track and revoke)
    - Implement rate limiting on your API endpoints
  </Accordion>

  <Accordion title="Validate and Sanitize Inputs" icon="filter">
    If your API accepts user input, validate it:

    - Check for SQL injection attempts
    - Validate data types and formats
    - Limit string lengths
    - Sanitize special characters
    - Use parameterized queries in your API
  </Accordion>

  <Accordion title="Limit Scope and Permissions" icon="user-shield">
    - Grant minimum necessary permissions to API credentials
    - Use read-only keys for lookup actions
    - Restrict write permissions to specific endpoints
    - Implement IP whitelisting if possible
    - Monitor for unusual activity
  </Accordion>

  <Accordion title="Handle Sensitive Data Carefully" icon="eye-slash">
    - Don't log sensitive data in action responses
    - Mask credit card numbers, SSNs, passwords in responses
    - Comply with PCI DSS, HIPAA, GDPR as applicable
    - Encrypt data at rest and in transit
    - Don't store unnecessary PII
  </Accordion>

  <Accordion title="Monitor and Audit" icon="chart-line">
    - Log all custom action executions
    - Monitor for failed authentication attempts
    - Track unusual patterns (high volume, errors)
    - Set up alerts for security events
    - Review logs regularly
    - Maintain audit trail for compliance
  </Accordion>
</AccordionGroup>

## Troubleshooting

<AccordionGroup>
  <Accordion title="401 Unauthorized Error" icon="lock-open">
    **Symptom:** Action fails with 401 error

    **Causes:**
    - Invalid credentials
    - Expired token
    - Wrong authentication type
    - Token missing "Bearer" prefix

    **Solutions:**
    - Verify credentials are correct and current
    - Check authentication type matches API requirements
    - For Bearer auth, ensure token format is correct
    - Test endpoint with Postman using same credentials
    - Check if token has expiration (refresh if needed)
    - Verify API key has necessary permissions
  </Accordion>

  <Accordion title="404 Not Found Error" icon="magnifying-glass">
    **Symptom:** Action returns 404

    **Causes:**
    - Incorrect endpoint URL
    - Resource doesn't exist
    - Template variable evaluated to invalid value
    - API endpoint changed/deprecated

    **Solutions:**
    - Verify endpoint URL is correct
    - Test URL with static values first
    - Check template variables populate correctly
    - Verify resource exists (test with known IDs)
    - Check API documentation for endpoint changes
  </Accordion>

  <Accordion title="500 Internal Server Error" icon="server">
    **Symptom:** API returns 500 error

    **Causes:**
    - Bug in your API
    - Database error
    - Invalid request format
    - Missing required fields

    **Solutions:**
    - Check your API server logs
    - Verify request format matches API expectations
    - Ensure all required fields are included
    - Test with minimal request to isolate issue
    - Contact your API team for investigation
  </Accordion>

  <Accordion title="Request Timeout" icon="clock">
    **Symptom:** Action fails with timeout

    **Causes:**
    - API responding slowly
    - Database query taking too long
    - Network issues
    - Endpoint unreachable

    **Solutions:**
    - Optimize API response time
    - Increase timeout setting if appropriate
    - Check network connectivity
    - Verify endpoint is reachable
    - Consider making operation asynchronous (webhook)
    - Cache frequently accessed data
  </Accordion>

  <Accordion title="Template Variables Not Populating" icon="brackets-curly">
    **Symptom:** Variables show as {{ variable }} in requests

    **Causes:**
    - Incorrect variable syntax
    - Variable doesn't exist
    - Field not populated in contact record

    **Solutions:**
    - Verify exact variable name: `{{ contact.email }}` not `{{ email }}`
    - Check contact record has field populated
    - Use defaults: `{{ contact.email | default("unknown") }}`
    - Test with contact record that has all fields
    - Review available variables documentation
  </Accordion>

  <Accordion title="Invalid JSON Response" icon="code">
    **Symptom:** Action succeeds but agent can't use data

    **Causes:**
    - API returning non-JSON response
    - Malformed JSON
    - Response wrapped in unexpected format

    **Solutions:**
    - Verify API returns valid JSON
    - Check Content-Type header in response
    - Use JSON validator on response
    - Check for HTML error pages being returned
    - Verify endpoint URL is correct (not hitting error page)
  </Accordion>

  <Accordion title="CORS Errors" icon="globe">
    **Symptom:** Browser-based calls fail with CORS error

    **Note:** This typically only affects web widget calls, not phone calls

    **Solutions:**
    - Configure CORS headers on your API
    - Allow itellicoAI domain in CORS policy
    - For testing, use phone calls not web calls
    - Consider using server-side proxy
  </Accordion>
</AccordionGroup>

## Performance Optimization

<AccordionGroup>
  <Accordion title="Minimize Response Time" icon="gauge-high">
    Keep API responses under 2-3 seconds to maintain conversation flow:

    - Optimize database queries
    - Use caching where appropriate
    - Index database columns used in lookups
    - Return only needed fields (not full objects)
    - Consider async processing for slow operations
  </Accordion>

  <Accordion title="Reduce Payload Size" icon="compress">
    Smaller responses = faster processing:

    - Return only necessary fields
    - Use pagination for large datasets
    - Compress responses (gzip)
    - Avoid embedding large binary data
    - Use appropriate data types (don't stringify numbers)
  </Accordion>

  <Accordion title="Cache Frequently Accessed Data" icon="database">
    Cache data that doesn't change often:

    - Product catalogs
    - Pricing information
    - Configuration data
    - User permissions
    - Reference data

    Update cache when data changes, not on every request.
  </Accordion>

  <Accordion title="Use Appropriate HTTP Methods" icon="network-wired">
    - GET for reads (cacheable)
    - POST for creates (not idempotent)
    - PUT for full updates (idempotent)
    - PATCH for partial updates (more efficient)
    - DELETE for removals (idempotent)

    Using correct methods enables better caching and performance.
  </Accordion>

  <Accordion title="Implement Rate Limiting" icon="gauge">
    Protect your API from overload:

    - Set reasonable rate limits
    - Return 429 when limit exceeded
    - Include Retry-After header
    - Implement exponential backoff
    - Monitor for abuse
  </Accordion>
</AccordionGroup>
