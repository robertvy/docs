## Introduction

Designing a strong prompt (instructions) is how you turn a generic voice model into a reliable AI teammate. This comprehensive guide walks through how the itellicoAI agent runtime interprets your instructions, the structure we recommend, and repeatable patterns you can use for different use cases.

<Tip>
This is an advanced guide. If you're just getting started, see [Instructions Basics](/build/conversation/instructions) first.
</Tip>

## Quick Start: The 5-Minute Prompt Formula

New to writing agent prompts? Start here. This formula gets you from blank page to working agent in minutes.

<Steps>
  <Step title="Who is your agent? (30 seconds)">
    Give your agent a clear identity and personality.

    ```
    You are [Name], a [role] at [Company].
    Your personality is [trait 1], [trait 2], and [trait 3].
    ```

    **Example:**
    ```
    You are Sarah, a Customer Success Manager at TechCo.
    Your personality is helpful, patient, and knowledgeable.
    ```
  </Step>

  <Step title="What's the goal? (30 seconds)">
    Define the specific, measurable outcome.

    ```
    Your objective is to [specific measurable outcome].
    ```

    **Example:**
    ```
    Your objective is to schedule product demos with qualified prospects
    who have budget and timeline.
    ```
  </Step>

  <Step title="How does the conversation flow? (3 minutes)">
    Map out the key steps in order.

    **Standard Flow:**
    1. **Greeting** - Confirm identity
    2. **Permission** - Ask if they have time
    3. **Discovery** - Understand their needs
    4. **Value** - Share how you can help
    5. **Action** - Book meeting/close sale
    6. **Closing** - Confirm next steps

    **Write it in your prompt like this:**
    ```
    Conversation Flow:
    1. Greeting - Confirm identity and thank them for their interest
    2. Permission - Ask if they have 5 minutes to talk
    3. Discovery - Understand their needs with open-ended questions
    4. Value - Share how you can help based on what you learned
    5. Action - Book meeting or close the sale using the appropriate action
    6. Closing - Confirm next steps and thank them for their time
    ```
  </Step>

  <Step title="What are the guardrails? (1 minute)">
    Define what your agent should NEVER do and when to escalate.

    **Write it in your prompt like this:**
    ```
    Guardrails - Never:
    • Discuss pricing (transfer to sales team)
    • Make promises about features or timelines
    • Share customer data or account details
    • Continue if customer is hostile (end call politely)

    Transfer to a human when:
    • Customer requests to speak with a manager
    • Technical issue beyond your knowledge
    • Customer is frustrated after 2 attempts to help
    ```
  </Step>

  <Step title="What if they object? (1 minute)">
    Prepare for common objections with exact responses.

    ```
    Objection Handling:
    • "Not interested" → [Your response]
    • "Too expensive" → [Your response]
    • "Need to think about it" → [Your response]
    ```

    **Example:**
    ```
    Objection Handling:
    • "Not interested" → "I understand. Can I ask what changed since you requested info?"
    • "Too expensive" → "I hear you. Let's discuss what features matter most to your budget."
    • "Need to think about it" → "Of course. What specific concerns should I address?"
    ```
  </Step>
</Steps>

### Key Writing Principles

Follow these three principles to make your prompts more effective:

<AccordionGroup>
  <Accordion title="Write Like You Talk" icon="comments">
    Voice agents need conversational language, not corporate speak.

    **Avoid:**
    ```
    "I am contacting you regarding..."
    "Per our previous correspondence..."
    ```

    **Better:**
    ```
    "I'm calling about..."
    "Following up on..."
    ```

    Test: Read it aloud. If it sounds stiff, simplify it.
  </Accordion>

  <Accordion title="Give Exact Words" icon="quote-left">
    Don't describe what to say - write the actual words.

    **Avoid:**
    ```
    "Introduce yourself professionally"
    "Explain the benefits"
    ```

    **Better:**
    ```
    Say: "Hi, this is Alex from TechCo"
    Say: "This saves you 10 hours per week on reporting"
    ```

    The more specific, the more consistent your agent will be.
  </Accordion>

  <Accordion title="Use If/Then Branching" icon="code-branch">
    Handle different scenarios with conditional logic.

    ```
    If customer says yes → Continue to step 2

    If customer says no → Ask: "When would be better?"

    If customer is hostile → Say: "I apologize for
    the inconvenience. Let me transfer you to a
    manager who can help."
    ```

    Plan for success, objections, and edge cases.
  </Accordion>
</AccordionGroup>

### Quick Tips for Better Prompts

<AccordionGroup>
  <Accordion title="Guardrails" icon="shield-halved">
    Be explicit about what the agent should never do.

    **Examples:**
    ```
    Guardrails - Never:
    - Make promises about features in development
    - Discuss competitor products
    - Provide legal or medical advice
    - Share pricing not in the approved price list
    - Continue if customer asks to be removed from list
    ```

    Clear guardrails prevent costly mistakes.
  </Accordion>

  <Accordion title="Natural transitions" icon="arrow-right-arrow-left">
    Help the agent move smoothly between conversation stages.

    **Examples:**
    ```
    "Now that we've covered X, let me share Y..."
    "Great question. Before I answer that, can I ask..."
    "That makes sense. Based on what you've told me..."
    "Perfect. So the next step would be..."
    ```

    Transitions make conversations feel natural, not scripted.
  </Accordion>

  <Accordion title="Confirmation protocols" icon="circle-check">
    Always verify critical information before taking action.

    **Examples:**
    ```
    Before booking:
    "Let me confirm - that's Tuesday, March 15th at 2 PM Eastern. Is that correct?"

    Before transferring:
    "Just to make sure I have this right, you need help with [issue]. Correct?"

    Before ending:
    "Before we wrap up, is there anything else I can help with today?"
    ```

    Reduces errors and builds trust.
  </Accordion>

  <Accordion title="Graceful exits" icon="door-open">
    Plan for every type of "no" with dignity.

    **Examples:**
    ```
    Not interested:
    "I completely understand. Thanks for your time, and feel free to reach
    out if anything changes."

    Wrong person:
    "No problem. Do you know who I should speak with about [topic]?"

    Bad timing:
    "I appreciate you letting me know. When would be a better time to call back?"
    ```

    How you end matters as much as how you start.
  </Accordion>

  <Accordion title="Progressive disclosure" icon="layer-group">
    Don't ask for everything upfront - build rapport first.

    **Bad:**
    ```
    "Can I get your name, email, phone number, company, role, and budget?"
    ```

    **Better:**
    ```
    1. Start with name
    2. Build rapport and establish value
    3. Ask for email (for confirmation)
    4. Naturally discover other details through conversation
    ```

    People share more when they understand why you're asking.
  </Accordion>
</AccordionGroup>

## Understanding the Instruction Panel

Inside **Agents → Abilities → Instructions** you provide a single block of text. The platform sends this block to the LLM on every turn together with:

- **Structured context** - Contact data, recent action results
- **Knowledge Base snippets** - Retrieved content relevant to the conversation
- **Real-time transcript** - The ongoing conversation history

### What the Agent Already Knows

The platform automatically provides:
- Full conversation history
- Available actions and their parameters
- Knowledge base content (when retrieved)
- Current date, time, and timezone

**Available via variables** (only when you reference them and contact exists in Contacts):
- Contact information: `{{ contact.first_name }}`, `{{ contact.email }}`, `{{ contact.phone }}`
- Context data: Dynamic values from API integrations

**Don't repeat this information** in your instructions. Focus on behavioral rules, decision-making logic, and business-specific guidance.

## Before You Write Anything

Collect these inputs so the prompt speaks to a specific scenario instead of a generic persona:

<Steps>
  <Step title="Define Call Objective">
    What should the caller achieve by the end of the conversation?

    Examples:
    - Get answer to product question
    - Book an appointment
    - Qualify as sales lead
    - Resolve technical issue
  </Step>

  <Step title="Identify Must-Have Data">
    What fields are needed for downstream actions?

    Examples:
    - Email address (for confirmations)
    - Policy number (for lookups)
    - Pain point (for qualification)
    - Preferred date/time (for booking)
  </Step>

  <Step title="List Available Actions">
    What actions can the agent trigger?

    Examples:
    - Transfer to Support
    - Book Appointment
    - Send Follow-up Email
    - Update CRM

    Include failover plans for each.
  </Step>

  <Step title="Map Knowledge Sources">
    Which knowledge collections are attached?

    Examples:
    - Product FAQ
    - Return Policies
    - Technical Troubleshooting
    - Pricing Information

    When should each be referenced?
  </Step>

  <Step title="Define Guardrails">
    What topics must be avoided? When to escalate?

    Examples:
    - Never discuss competitor products
    - Never promise refunds (transfer instead)
    - Escalate legal threats immediately
    - Don't diagnose hardware (schedule technician)
  </Step>
</Steps>

**Documenting these inputs first** keeps your prompt short, specific, and easier to maintain.

## Recommended Prompt Structure

Use this structure as a template:

| Section | Purpose | Tips |
|---------|---------|------|
| **Role & Tone** | Tell the LLM who it is representing and how it should sound | Include brand tone, languages allowed, pronunciation notes |
| **Primary Goal** | Explain the desired outcome for the caller | Use bullets or numbered lists for clarity |
| **Conversation Flow** | Lay out the core steps in order | Reference prerequisites, loops, exit criteria |
| **Information Capture** | Call out required fields and confirmation steps | Highlight how to verify accuracy before proceeding |
| **Knowledge Usage** | Instruct when to pull in knowledge base facts | Mention collection names or keywords to search for |
| **Action Triggers** | Describe when to run each action and with which inputs | Use explicit action names that match the configuration UI |
| **Escalations & Guardrails** | Define forbidden topics and when to hand off or end calls | Pair rules with the relevant transfer/end action |
| **Closure & Follow-up** | Spell out how to wrap up the call and confirm next steps | Include any follow-up messaging you promise the caller |

<Tip>
Keeping these sections in a consistent order helps later editors understand the prompt quickly.
</Tip>

## Reusable Patterns

<AccordionGroup>
  <Accordion title="Pattern: Sequencing" icon="list-ol">
    ```
    Follow these steps in order:
    1. [First step]
    2. [Second step]
    3. [Third step]

    If interrupted, return to the first unfinished step before moving on.
    ```

    **Use when:** You need strict conversation flow
  </Accordion>

  <Accordion title="Pattern: Acknowledgement & Empathy" icon="heart">
    ```
    When customers express frustration:
    - Acknowledge feelings: "I understand how frustrating that must be"
    - Apologize when appropriate: "I'm sorry you're experiencing this"
    - Take ownership: "Let me see what I can do to help"
    - Avoid defensive language
    ```

    **Use when:** Tone matters for customer satisfaction
  </Accordion>

  <Accordion title="Pattern: Action Safety Nets" icon="shield">
    ```
    If the [Action Name] action fails:
    1. Apologize sincerely: "I apologize, I'm having trouble with that"
    2. Explain what went wrong briefly
    3. Offer alternative: [describe alternative]
    4. If alternative also fails, trigger [Fallback Action]
    ```

    **Use when:** Actions might fail and you need graceful recovery
  </Accordion>

  <Accordion title="Pattern: Multi-Language Support" icon="globe">
    ```
    Languages allowed: English, Spanish, French

    If the caller speaks Spanish:
    - Respond in Spanish
    - Use formal addressing (usted)

    If the caller speaks French:
    - Respond in French
    - Use formal addressing (vous)
    ```

    **Use when:** Serving multilingual customers
  </Accordion>

  <Accordion title="Pattern: Compliance Statements" icon="scale-balanced">
    ```
    Important compliance rules:
    - NEVER promise service availability outside business hours
    - NEVER discuss pricing not listed in the current price sheet
    - NEVER share customer data without verification
    - ALWAYS disclose that calls may be recorded (already in greeting)

    If asked about regulated topics, say: "I need to transfer you to [Department] for that information."
    ```

    **Use when:** Legal/regulatory requirements exist
  </Accordion>
</AccordionGroup>

## Referencing Actions

<Warning>
Always reference actions by the **exact name** configured in the UI. The agent runtime exposes a function call with that name, so ambiguous wording can break execution.
</Warning>

### Example: Booking Action

```
When the caller confirms they want to schedule, call the `Schedule Dental Checkup` booking action with:
- `date` and `time` gathered during the conversation
- `timezone` from `{{ contact.timezone | default("America/Los_Angeles") }}`
- Ask for email if `contact.email` is missing; otherwise use contact email

If the action returns an error:
1. Apologize: "I'm sorry, I'm having trouble booking that time slot"
2. Offer two alternative slots from the same day
3. Retry once with the alternative
4. If the second attempt fails, trigger the `Transfer to Front Desk` action
```

### Example: Transfer Action

```
Use the `Transfer to Tier 2 Support` action when:
- Issue requires escalation beyond your capabilities
- Customer explicitly requests human agent
- Technical issue unresolved after troubleshooting
- Legal or billing questions arise

Before transferring:
1. Explain why: "I'm going to connect you with a specialist who can help"
2. Summarize what you've discussed so they don't have to repeat themselves
3. Thank them for their patience
4. Then trigger the transfer action
```

## Using Knowledge Effectively

### Mention Collection Names

```
Knowledge Base Usage:

Use the `Product FAQ` collection when:
- Callers ask about product features
- Questions about compatibility
- General "how does it work" questions

Use the `Return Policies` collection when:
- Callers ask about returns or refunds
- Questions about warranty coverage
- Exchange processes

Use the `Troubleshooting Guide` collection when:
- Technical issues reported
- Error messages mentioned
- Performance problems described
```

### Instruct How to Cite

```
When using knowledge base information:
- Summarize policies in your own words
- Don't read verbatim legal text
- Use conversational language
- Confirm understanding: "Does that answer your question?"
```

### Handle Conflicting Information

```
Priority when information conflicts:
1. Live context data ({{ context.current_offer }})
2. Recently updated knowledge bases (marked with dates)
3. General knowledge base content

If knowledge base contradicts information from dynamic context fields, prioritize the live context data.
```

## Variables and Conditional Content

### Jinja Templating Basics

Personalize instructions with dynamic values:

```jinja
The caller's name is {{ contact.first_name | default("there") }}.
Their account type is {{ contact.account_type | default("standard") }}.
Their timezone is {{ contact.timezone | default("America/New_York") }}.
```

### Conditional Logic

```jinja
{% if contact.account_type == "VIP" %}
IMPORTANT: This is a VIP customer. Acknowledge their status:
"Thank you for being a valued VIP customer, we really appreciate your business."
Prioritize their requests and offer premium solutions first.
{% endif %}

{% if contact.language == "es" %}
Respond in Spanish and use formal addressing (usted).
{% else %}
Respond in English conversationally.
{% endif %}
```

### Using Defaults

```jinja
Timezone: {{ contact.timezone | default("America/New_York") }}
Organization: {{ organization.name | default("our company") }}
Offer: {{ context.current_offer | default("our standard services") }}
```

<Tip>
Use conditionals sparingly and favor defaults to avoid rendering empty phrases. Testing prompts with dummy context values helps catch mistakes.
</Tip>

    You are "Riley", the friendly voice of {{ organization.name }}. Sound warm, concise, and professional.

    Goal:
    1. Identify the caller's product and issue category
    2. Capture name, callback number, and best email
    3. Resolve basic questions using the Knowledge Base. If unable, escalate

    Flow:
    1. Greet the caller, confirm how you can help
    2. Gather account or order identifier; verify spelling
    3. Diagnose using the `Product FAQ` collection. Summarize solutions plainly
    4. If unresolved after two attempts or if caller requests a human, trigger `Transfer to Tier 2`

    Actions:
    - `Transfer to Tier 2`: use when policy requires human follow-up or caller insists
    - `End Standard Call`: use only after confirming next steps and asking if anything else is needed

    Guardrails:
    - Never promise refunds or credits; instead, explain policy and escalate
    - If caller mentions emergencies or legal threats, escalate immediately

    Close the call by recapping the resolution or next steps and thanking them for choosing {{ organization.short_name }}.
    ```

  <Tab title="Outbound Follow-up">
    ```
    Role: Outbound concierge calling warm leads who requested a consultation. Be energetic, keep responses under 25 seconds.

    Target Outcome: Book a demo within 7 days or confirm lack of interest.

    Steps:
    1. Confirm you're speaking with the correct person using {{ contact.first_name }}
    2. Remind them of their original request
    3. Qualify interest with two discovery questions before offering times
    4. Offer two specific time slots from available inventory
    5. Trigger `Book Demo` action once time confirmed
    6. If they decline, mark reason in notes and use `End Prospect Call`

    If voicemail detected:
    "Hi {{ contact.first_name }}, this is [Name] from {{ organization.name }}. You requested information about [product]. I'll try you again tomorrow, or you can reach us at [number]. Thanks!"

    Objection handling:
    - "Too busy" → Offer specific short time slots, emphasize 15-minute duration
    - "Not interested" → Ask what changed, attempt one soft requalification
    - "Already chose competitor" → Thank them, ask for feedback, mark lost in CRM
    ```
  </Tab>

  <Tab title="Technical Support">
    ```
    You are a {{ organization.name }} technical support specialist helping with {{ product.name }}.

    Primary Goal: Resolve technical issues efficiently and professionally.

    Conversation Flow:
    1. Greet and confirm the technical issue
    2. Gather essential info (product version, error message, steps attempted)
    3. Search `Troubleshooting Guide` knowledge collection
    4. Walk through solution step-by-step (one step at a time)
    5. Confirm each step completed before moving to next
    6. Verify issue resolved
    7. Ask if anything else needed

    Troubleshooting Protocol:
    - Ask one diagnostic question at a time
    - Confirm understanding of responses
    - Use simple, non-technical language unless customer is technical
    - If solution requires > 5 steps, offer to schedule technician visit

    Actions:
    - `Schedule Technician Visit`: Use when remote troubleshooting insufficient
    - `Create Support Ticket`: Use for issues requiring engineering investigation
    - `Transfer to Level 3`: Use for complex hardware or network issues

    Knowledge Usage:
    - Reference `Troubleshooting Guide` for step-by-step procedures
    - Check `Known Issues` collection for active problems
    - Use `Product Documentation` for feature explanations

    Guardrails:
    - Never guess at solutions - use knowledge base or escalate
    - Never ask customer to open physical devices (safety risk)
    - If frustrated after 10 minutes, offer escalation proactively
    ```
  </Tab>

    Objective: Qualify leads and route to appropriate sales rep within 5 minutes.

    Qualification Criteria:
    - Budget: > $10k annually (ask: "What budget range have you allocated?")
    - Timeline: Decision within 90 days (ask: "When are you looking to implement?")
    - Authority: Can influence or make purchase decisions (ask: "Who else is involved in the decision?")
    - Need: Clear pain point we can solve (ask: "What challenges are you trying to solve?")

    Conversation Flow:
    1. Warm greeting, thank them for interest
    2. Ask discovery questions naturally (don't interrogate)
    3. Listen actively, acknowledge responses
    4. Qualify against criteria (all 4 must be met for "qualified")

    - **Qualified** (all 4 criteria met): Book meeting, mark as "Hot Lead" in CRM
    - **Partially Qualified** (2-3 criteria): Send resources, mark as "Warm Lead", schedule follow-up
    - **Unqualified** (0-1 criteria): Politely disengage, mark as "Cold", add to newsletter

    Actions:
    - `Book Sales Call`: For qualified leads only, get email and phone confirmation
    - `Send Resources Email`: For partially qualified, send relevant case studies
    - `Update CRM`: Always update with qualification status and notes

    Tone: Professional, consultative, not pushy. Build rapport but stay efficient.

    If lead asks about pricing:
    "Pricing varies based on your specific needs. Once I understand your situation better, I can connect you with someone who can provide accurate pricing. May I ask a few questions about your requirements?"
    ```

    ```
    Escalation Triggers - Transfer Immediately When:

    URGENT (Transfer to Manager):
    - Legal language used ("lawyer", "sue", "legal action")
    - Threat of harm or violence
    - Data breach concerns
    - Major service outage impact

    HIGH PRIORITY (Transfer to Specialist):
    - Customer dissatisfaction after 2 resolution attempts
    - Request for supervisor or manager
    - Complex technical issue beyond knowledge base
    - Large account (> $50k annual value from {{ contact.account_value }})

    STANDARD (Complete Call):
    - General questions answerable from knowledge base
    - Standard troubleshooting
    - Information requests
    - Appointment booking/rescheduling
    ```
  </Accordion>

  <Accordion title="Response Pacing" icon="gauge">
    ```
    Response Guidelines:

    Length:
    - Keep responses under 30 seconds when possible
    - Break long explanations into chunks with confirmation checks
    - For multi-step processes, confirm after each step

    Pacing:
    - Speak at moderate pace (not rushed)
    - Pause briefly after asking questions
    - Use filler phrases naturally: "Let me check that for you", "One moment please"
    - Don't use excessive filler words (um, uh, like)

    Checking Understanding:
    After explaining something complex:
    - "Does that make sense so far?"
    - "Do you have any questions about that?"
    - "Would you like me to clarify anything?"
    ```
  </Accordion>

## Testing and Iteration

<AccordionGroup>
  <Accordion title="Test Methodology" icon="vial">
    <Steps>
      <Step title="Dry Run in Agent Tester">
        Test that agent follows the sequence, captures required data, and triggers the correct actions.

        Use the Test Call feature to simulate conversations.
      </Step>

      <Step title="Review Analytics">
        Check the Call Analysis panel to see if goals were met. Identify prompt gaps.

        Look for patterns in failed conversations.
      </Step>

      <Step title="Analyze Transcripts">
        Look for:
        - Hallucinations (making up information)
        - Missing disclaimers or guardrails
        - Overlong responses
        - Incorrect action triggers
        - Poor knowledge retrieval
      </Step>

      <Step title="Version Control">
        Before major changes:
        - Duplicate the agent for A/B testing
        - Paste the previous prompt into Notes with date
        - Document what changed and why
      </Step>
    </Steps>
  </Accordion>

  <Accordion title="Common Issues and Fixes" icon="wrench">
    | Issue | Symptom | Fix |
    |-------|---------|-----|
    | **Too Verbose** | Responses over 45 seconds | Add: "Keep responses under 25 seconds. Break long explanations into chunks." |
    | **Doesn't Use Actions** | Never triggers booking/transfer | Add explicit triggers with examples. Use exact action names. |
    | **Ignores Knowledge** | Answers without checking KB | Add: "Always check attached knowledge bases before answering." |
    | **Wrong Tone** | Too formal or too casual | Add tone examples and constraints. Use sample phrasing. |
    | **Interrupts Often** | Cuts off customers | Adjust VAD settings, not prompt. Or add: "Let customers finish speaking fully." |
    | **Doesn't Follow Flow** | Skips steps | Number steps clearly. Add: "Do not skip steps. Complete each before moving to next." |
    | **Poor Escalation** | Doesn't transfer when needed | Define explicit escalation triggers. Make rules unmistakable. |
  </Accordion>
</AccordionGroup>

## Best Practices Summary

<AccordionGroup>
  <Accordion title="Do" icon="check">
    - Be specific and explicit
    - Use exact action names from UI
    - Provide examples for complex scenarios
    - Test thoroughly after changes
    - Version control significant changes
    - Keep focused on behavioral rules
    - Use numbered lists for sequences
    - Define clear escalation criteria
  </Accordion>

  <Accordion title="❌ Don't" icon="xmark">
    - Write overly long prompts (> 1000 words)
    - Use vague instructions ("be helpful")
    - Include contradictory rules
    - Hardcode facts (use knowledge bases)
    - Forget to test edge cases
    - Make changes without backup
    - Include information agent already knows
    - Use ambiguous action references
    - Skip defining guardrails
  </Accordion>
</AccordionGroup>

## Ongoing Maintenance

Prompt engineering is **ongoing**. Schedule periodic reviews, especially after:

- Product launches
- Policy changes
- New feature releases
- Seasonal promotions
- Customer feedback indicating issues
- Analytics showing goal achievement drops

### Review Checklist

- [ ] Instructions still accurate for current business?
- [ ] Action names match current UI?
- [ ] Knowledge collections still relevant?
- [ ] Guardrails up to date with compliance?
- [ ] Tone still matches brand voice?
- [ ] Examples still applicable?
- [ ] Goal achievement rates acceptable?
